# 01 - 数据库脚本与表清单

## 1. 执行口径

当前项目未上线，没有客户数据，`sql/postgresql/product_capability.sql` 可以直接重构。

允许：

```text
删除旧的一期不合理 pc_* 表
重命名旧表
重写字段
重写菜单初始化数据
重写字典初始化数据
重建开发库
```

不需要为了旧的一期骨架做兼容迁移。

## 2. 表分组

表分组按业务依赖排序：

```text
基础资料
  -> 工程配置
  -> 产品配置中心
```

### 基础资料

建议表：

```text
pc_product_category        产品分类
pc_material                 统一物料资料 / 货品主数据
pc_material_attribute       物料属性值
pc_base_attribute           属性定义
pc_unit                     单位
pc_fabric_series           面料系列
pc_fabric_profile          面料扩展信息
pc_component               组件 / 辅材
pc_component_item          组件包明细
pc_media_asset             资料资产
pc_media_binding           资料绑定
```

说明：

```text
pc_material 是基础信息的统一大主表，承载面料、铝材、电机、拉珠、遥控器、太阳能板、配件、安装件、包装件、标准成品等。
不要为铝材、系统、配件、包装分别拆一套主数据表，它们通过 material_type 和 pc_material_attribute 区分。
面料因为有系列、门幅、厚度等高频工程配置属性，保留 pc_fabric_series + pc_fabric_profile。
pc_component / pc_component_item 只表达组件包或安装包由哪些物料组成，不替代物料主数据。
pc_material 必须保留 spec_summary，用于保存标准规格 + 高频补充属性摘要，减少列表、导出、工程预览多表联查。
基础资料不保存销售单价，不承载报价规则。
当前不做采购订单、库存流水、MRP，但物料主数据必须预留采购/库存/使用单位、供应商、供应商料号等基础字段。
```

### 工程配置

建议表：

```text
pc_engineering_plan              工程方案
pc_engineering_plan_version      工程方案版本
pc_engineering_item              工程构成项
pc_engineering_item_scope        构成项可选范围
pc_engineering_rule              工程能力规则
pc_engineering_output_rule       工程带出规则
pc_standard_sku_engineering      标品固定工程配置
pc_engineering_check_case        工程配置检查样例
```

说明：

```text
工程配置只表达产品系列由哪些构成项组成、每个构成项可选哪些物料或组件包、什么条件下能选或不能选、会带出哪些组件包或资料。
工程配置不做裁切扣减、真实生产用量、库存扣减、采购计划和车间工艺。
```

### 产品与配置

建议表：

```text
pc_sales_product           销售产品 / 下单入口
pc_standard_sku            标品 SKU
pc_config_template         下单配置模板
pc_config_template_version 配置模板版本
pc_question_group          问题组
pc_config_question         配置问题
pc_config_option           配置答案
pc_config_rule             配置联动 / 可见性 / 校验规则
```

说明：

```text
pc_sales_product 后续绑定工程方案版本。
配置问题和答案后续映射工程构成项与可选范围，不再自己承载核心工程规则。
```

### 价格、发布、订单快照

建议表：

```text
pc_price_plan              价格方案
pc_price_plan_version      价格方案版本
pc_price_rule_item         价格规则项 / Price Grid / 加价规则
pc_publish_check_result    发布检查结果
pc_publish_approval        发布审批
pc_product_publish_package 发布包
pc_product_snapshot        订单产品能力快照
pc_product_sync_outbox     同步日志 / Outbox
pc_import_batch            导入批次
pc_import_row_issue        导入行问题
```

## 3. 旧表处理建议

| 旧表 | 处理 |
| --- | --- |
| `pc_category` | 可重命名为 `pc_product_category`，或保留但文档统一叫产品分类 |
| `pc_material` | 可重构为 `pc_item_master`，或短期保留并补面料扩展表 |
| `pc_component` | 保留，补组件包明细能力 |
| `pc_media_asset` | 保留，继续引用 OSS |
| `pc_media_binding` | 保留，扩展绑定对象 |
| `pc_product_model` | 可重命名为 `pc_sales_product` |
| `pc_sales_variant` | 可拆成销售变体和标品 SKU，第一版建议明确命名 |
| `pc_config_*` | 保留但字段按下单配置重整 |
| `pc_price_*` | 保留，补 Price Grid 和试算字段 |
| `pc_publish_*` | 保留，发布检查要从骨架变成真实检查 |
| `pc_product_snapshot_instance` | 建议改名为 `pc_product_snapshot`，表达订单快照更直观 |

## 4. 数据库脚本第一批任务

```text
1. 备份当前 product_capability.sql
2. 重整 pc_* 表命名和注释
3. 补 pc_material.spec_summary、采购/库存/使用单位、供应商料号等基础字段
4. 补工程方案、构成项、可选范围、工程规则、带出规则、标品固定工程配置表
5. 重写基础信息和工程配置菜单初始化数据
6. 重写基础信息和工程配置字典初始化数据
7. 在临时 PostgreSQL 容器执行脚本
8. 用开发库重建验证
```

## 5. 注意

当前不是线上迁移，所以可以直接改 SQL。

但每次改 SQL 都要能回答：

```text
这个表属于哪个模块？
这个表被哪个页面维护？
这个表被哪个试算器或发布检查使用？
订单快照是否需要保存它的版本？
```
