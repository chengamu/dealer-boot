# 00 - 总原则与执行顺序

## 1. 当前前提

```text
项目未上线
没有客户数据
以前第一期做过的产品配置中心可以改动、删除、重建
sql/postgresql/product_capability.sql 可以直接修改
开发库数据库可以配合重建
```

这次按“首次正式编码前重构”处理，不按线上平滑迁移处理。

## 2. 总目标

一期不是完整 ERP，也不是完整 MRP。

一期目标是：

```text
基础资料能统一维护未来 ERP 也要消费的物料、属性、单位、组件包和资料资产
技术科能维护产品系列的工程方案、工程构成项、可选范围、能力规则和带出规则
工程配置能预览定制和标品在不同选择下能选什么、不能选什么、带出什么
产品配置中心先保留，后续作为客户下单问题展示层消费工程配置
```

## 3. 业务依赖顺序

业务上必须先有基础资料，再有工程配置，最后产品配置中心再消费工程配置：

```text
基础资料
  -> 工程配置
  -> 产品配置中心
```

解释：

```text
基础资料：维护统一物料资料、物料属性、单位、组件包、资料资产，是未来 ERP 主数据底座。
工程配置：基于基础资料定义产品系列的工程构成、可选范围、限制规则、组件/资料带出和预览检查。
产品配置中心：后续把工程配置映射成客户可见问题和答案；当前不继续扩展。
```

产品配置中心不能脱离工程配置独立加深。否则只会得到问题/答案 CRUD，无法表达产品系列真正的工程构成和限制。

## 4. 实施执行顺序

```text
数据库脚本
  -> 后端 domain / mapper / service / controller
  -> 前端 api / 页面 / 路由 / i18n
  -> 试算器和发布检查
  -> 订单快照闭环
```

不要从前端页面开始硬拼。

## 5. 验收口径

每个模块必须至少满足：

```text
数据库表能解释业务对象
后端接口有真实读写或真实计算
前端不靠假数据
权限和菜单能进入
关键动作能在浏览器完成
失败原因可见
```

只有 CRUD 的模块，必须能新增、编辑、停用、查询、引用检查。

试算器模块，必须返回后端计算结果和命中明细。

## 6. 仍然复用的基础能力

```text
OSS 文件能力
登录和权限
菜单和按钮权限
系统字典
i18n
UTC 时间规则
审计字段
Element Plus 管理端风格
```

这些不要重做。
