# 00 - 总原则与执行顺序

## 1. 当前前提

```text
项目未上线
没有客户数据
以前第一期做过的产品配置中心可以改动、删除、重建
sql/postgresql/product_capability.sql 可以直接修改
开发库数据库可以配合重建
```

这次按“首次正式编码前重构”处理，不按线上平滑迁移处理。

## 2. 总目标

一期不是完整 ERP，也不是完整 MRP。

一期目标是：

```text
技术科能维护工程模板、尺寸限制、BOM 规则、组件包、包装规则
产品能维护客户下单看到什么、能选什么
基础资料能统一维护面料、物料、附件和单位
订单能基于已发布配置生成配置快照、价格快照、BOM 快照、资料快照
```

## 3. 执行顺序

```text
数据库脚本
  -> 后端 domain / mapper / service / controller
  -> 前端 api / 页面 / 路由 / i18n
  -> 试算器和发布检查
  -> 订单快照闭环
```

不要从前端页面开始硬拼。

## 4. 验收口径

每个模块必须至少满足：

```text
数据库表能解释业务对象
后端接口有真实读写或真实计算
前端不靠假数据
权限和菜单能进入
关键动作能在浏览器完成
失败原因可见
```

只有 CRUD 的模块，必须能新增、编辑、停用、查询、引用检查。

试算器模块，必须返回后端计算结果和命中明细。

## 5. 仍然复用的基础能力

```text
OSS 文件能力
登录和权限
菜单和按钮权限
系统字典
i18n
UTC 时间规则
审计字段
Element Plus 管理端风格
```

这些不要重做。
