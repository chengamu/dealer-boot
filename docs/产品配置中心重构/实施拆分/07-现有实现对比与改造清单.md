# 07 - 现有实现对比与改造清单

## 1. 当前结论

当前实现不是空白，但还没有到“技术科可维护、订单可放心消费”的程度。

现有系统更接近：

```text
基础资料 CRUD 骨架
+ 产品配置问题 / 答案 CRUD 骨架
+ 轻量配置求值
+ 轻量价格 / 发布 / 快照预留
```

下一阶段不继续扩大产品配置中心，而是先补齐：

```text
基础信息
  -> 工程配置
```

产品配置中心先保留现有入口，后续只消费工程配置结果，不再把工程能力、构成项、物料带出关系继续塞进配置答案里。

---

## 2. 现在已实现的部分

### 2.1 基础资料

当前已经有可继续改造的基础：

```text
产品分类
物料资料
物料属性
单位
面料系列
面料资料
组件包
组件明细
资料资产
资料绑定
```

这些能力方向是对的，但还不够完整：

```text
物料资料还没有完全按未来 ERP 主数据口径收敛
物料规格摘要字段需要明确
物料类型和属性模板需要更清楚
采购 / 库存 / 用量单位要保留基础字段
供应商物料号、标准规格、补充规格信息要能被搜索、列表和导出直接使用
```

### 2.2 产品配置

当前已有：

```text
销售产品
配置模板
模板版本
问题组
配置问题
配置答案
配置求值器
```

但当前产品配置只适合表达：

```text
下单时问什么
客户能选什么
答案来自哪里
轻量显示 / 禁用 / 必填
```

它不适合继续承载：

```text
产品由哪些工程构成项组成
什么条件下带出哪些组件
什么尺寸 / 面料 / 系统组合不可做
标品固定工程构成
技术科维护的工程方案检查
```

这些必须进入工程配置。

## 3. 基础信息差距清单

| 能力 | 当前状态 | 缺口 | 改造方向 |
| --- | --- | --- | --- |
| 统一物料资料 | 已有 `pc_material` | 语义还要统一为原材料、组件、系统件、配件、包装、标品成品的一张主表 | 保留一张大表，使用 `material_type` 区分类型 |
| 规格字段 | 有标准规格雏形 | 缺少明确的规格摘要口径 | 增加或统一 `primary_spec`、`spec_summary` |
| 物料属性 | 已有属性表方向 | 属性模板和搜索展示不足 | `pc_material_attribute` 存结构化属性，`spec_summary` 存列表摘要 |
| 单位 | 已有单位管理方向 | 采购单位、库存单位、用量单位要为未来 ERP 保留 | 物料主表保留 `purchase_unit_code`、`inventory_unit_code`、`usage_unit_code` |
| 供应商信息 | 有部分字段 | 供应商编码、名称、供应商物料号需要成为物料基础字段 | 保留 `supplier_code`、`supplier_name`、`vendor_item_no` |
| 面料资料 | 已有面料系列 / profile | 双面料、厚度组合等要明确边界 | 原始厚度、门幅放基础资料，组合是否可用由工程配置判断 |
| 组件包 | 已有组件和明细 | 需要更像“预配置零件包” | 继续保留 `pc_component`、`pc_component_item`，工程配置引用组件包 |
| 资料资产 | 已有资产和绑定 | 需要继续复用 OSS，不重做附件系统 | 资料资产只做 metadata 和绑定，文件本体仍走 OSS |

基础信息的核心原则：

```text
物料资料只维护“是什么”
工程配置维护“在什么产品、尺寸、选项条件下怎么用”
产品配置维护“客户下单时怎么问、怎么选”
```

## 4. 工程配置缺口清单

当前最缺的是工程配置模块。它不是生产工艺，也不是 ERP 生产用量模块。

工程配置要解决：

```text
一个产品系列 / 销售产品由哪些工程构成项组成
每个构成项可从哪些物料、组件包、属性或固定值里选择
不同尺寸、面料、系统、控制方式下哪些选项可用或禁用
哪些选择会带出哪些组件包或物料引用
标品 SKU 的固定工程构成如何快速维护
技术科如何预览和检查一套工程方案
```

建议新增或补齐：

| 能力 | 说明 |
| --- | --- |
| 工程方案 | 按产品系列或销售产品维护工程配置方案，例如蜂巢帘、卷帘、柔纱帘 |
| 工程方案版本 | 草稿、启用、停用，发布后给产品配置和订单使用 |
| 工程构成项 | 面料、第二面料、铝材、铝材颜色、系统、配件、安装零件包、包装方式 |
| 构成项可选范围 | 指定某个构成项能从哪些物料类型、系列、组件包、属性值里选择 |
| 工程能力规则 | 判断什么组合可做、不可做、需提示、需补件 |
| 工程带出规则 | 当选择某个系统、尺寸或面料时，带出组件包、配件、包装方式等 |
| 标品固定工程配置 | 大货标准 SKU 直接固定面料、系统、配件、包装，不要求客户逐项选择 |
| 工程配置预览 | 输入产品、尺寸、选择项，展示可选项、禁用项、带出项和缺失项 |
| 工程配置检查 | 检查工程方案是否缺构成项、缺可选范围、缺默认项、引用已停用资料 |

工程构成项建议：

```text
MAIN_FABRIC       主面料
SECOND_FABRIC     第二面料，可选，用于双面料组合
ALUMINUM          铝材
ALUMINUM_COLOR    铝材颜色
SYSTEM            系统，例如拉珠、电机、无拉、遥控、太阳能
ACCESSORY         公用配件
INSTALL_KIT       安装零件包
PACKAGING         包装方式
```

工程能力规则示例：

```text
同一系列允许双面料，但两块面料总厚度不能超过系列限制
某个具体面料超过宽度上限时不能选择电机
某个尺寸范围下不能选拉珠
宽度超过某个阈值时需要提示拆分或双电机方案
系统 = 电机时才显示遥控器 / 太阳能相关选择
包装方式根据成品尺寸匹配普通箱、长箱或特殊包装
```

这些规则不属于 ERP 生产工艺，不计算裁切扣减，不计算真实生产用量。

## 5. 产品配置中心怎么放

产品配置中心不删除，但要降级为下游消费层。

当前阶段处理方式：

```text
保留销售产品、配置模板、问题组、问题、答案、求值器
停止继续把组件、工程能力、尺寸可做性塞进配置答案
后续销售产品绑定工程方案版本
配置问题映射工程构成项
配置答案映射工程可选项或工程带出规则入口
配置求值器调用工程配置预览，返回客户可见问题、可选答案、禁用原因
```

也就是说：

```text
基础信息：维护物料和资料
工程配置：维护产品怎么组成、什么条件下怎么带出
产品配置：维护客户下单时怎么问
```

## 7. 第一批改造顺序

### Wave 1：基础信息补强

```text
1. 统一 pc_material 语义为物料主数据
2. 补齐 material_type、primary_spec、spec_summary、supplier、vendor_item_no、采购/库存/用量单位
3. 梳理 pc_material_attribute 的属性模板和列表展示字段
4. 面料系列继续承载门幅、厚度等原始资料
5. 组件包继续承载预配置零件包
```

### Wave 2：工程配置建模

```text
1. 新增工程方案和版本
2. 新增工程构成项
3. 新增构成项可选范围
4. 新增工程能力规则
5. 新增工程带出规则
6. 新增标品固定工程配置
```

### Wave 3：工程配置前后端闭环

```text
1. 工程方案 CRUD
2. 构成项和可选范围维护
3. 能力规则和带出规则维护
4. 工程配置预览
5. 工程配置检查
```

### Wave 4：产品配置桥接

```text
1. 销售产品绑定工程方案版本
2. 配置问题映射工程构成项
3. 配置答案映射工程可选项
4. 配置求值器调用工程配置预览
```

Wave 4 只做桥接，不扩价格、订单和 ERP。

## 8. 当前最大风险

```text
继续扩产品配置页面，会把工程能力塞成一堆配置答案，后续难维护。
继续把物料拆成多张类型表，会让未来采购、库存、工程引用变复杂。
继续使用旧的工程模块命名，会让技术科误以为这里要维护生产用量和工艺扣减。
不做工程配置，订单系统后面无法稳定判断“能不能做、带出什么、缺什么”。
不保留物料主数据 ERP 字段，后续采购、库存、成本会补得很痛苦。
```

所以本轮正确目标是：

```text
基础信息补成未来 ERP 可复用的主数据底座
工程配置补成技术科可维护的工程方案中心
产品配置中心暂时保留，后续消费工程配置
```
