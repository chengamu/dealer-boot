# 07 - 现有实现对比与改造清单

## 1. 当前实现结论

已经对当前 `base-boot` 里的产品配置中心实现做过对比。

当前不是空白，也不是完全没用；当前实现更像：

```text
共享产品能力中心 CRUD 骨架
+ 配置求值轻量校验
+ 价格计算轻量规则
+ 发布包轻量检查
+ 快照实例轻量保存
```

但它还不是海外定制订单需要的：

```text
基础资料
  -> 产品配置
  -> 工程模板
  -> BOM 规则
  -> 价格试算
  -> 发布检查
  -> 订单快照
```

所以改造方向不是“继续补 CRUD”，而是：

```text
保留可用底座
删除或重写泛化骨架
补工程中心和 BOM 主链路
把面料、附件、订单快照做成真实业务对象
```

## 3. 和目标方案的差异

| 目标能力 | 当前实现 | 差距 | 改法 |
| --- | --- | --- | --- |
| 面料设置 | 只有 `pc_material.material_type = FABRIC` 口径 | 没有面料系列、面料扩展属性、色卡资料绑定闭环 | 新增 `pc_fabric_series`、`pc_fabric_profile`，新增面料设置页面和接口 |
| 物料主数据 | 有 `pc_material` | 名称偏产品物料，未统一表达真实货品、成品 SKU、服务项 | 可重构为 `pc_item_master`，或短期保留并重写字段语义 |
| 组件包 | 有 `pc_component` | 没有组件包明细，不能表达安装包由多个物料组成 | 补 `pc_component_item`，组件支持 KIT |
| 附件资料 | 有 `pc_media_asset`、`pc_media_binding` | 方向正确，但要明确复用 OSS，不重做上传 | 保留，补 OSS 引用、版本、客户可见、订单快照规则 |
| 销售产品 | 有 `pc_product_model` | 更像泛产品模型，不够像客户下单入口 | 可重命名或语义重写为 `pc_sales_product` |
| 标品 SKU | 有 `pc_sales_variant` | 销售变体和标品 SKU 混在一起 | 第一版明确字段区分，后续可拆 `pc_standard_sku` |
| 配置模板 | 有模板、版本、问题、答案、规则 | 能录，但答案来源、适用范围、工程/BOM 绑定不足 | 补 sourceType/sourceRef、工程模板绑定、BOM 触发 |
| 配置求值 | 有 `ConfigEvaluationEngine` | 只做模板版本必填、问题必填、组件 JSON 解析 | 改成返回可见问题、禁用答案、资料、组件、工程输入、BLOCKER |
| 工程模板 | 无 | 技术科规则没有落点 | 新增工程模板、版本、尺寸限制、扣减、BOM 规则 |
| BOM 预览 | 无真实模块 | 当前只有 `componentJson` 摘要，不是生产 BOM | 新增 `BomPreviewEngine` 和 BOM 预览接口 |
| 价格中心 | 有价格方案、版本、规则项和 `PriceCalculationEngine` | 价格计算已有起点，但 Price Grid、Dealer 价、运费不完整 | 保留引擎，补规则类型、命中明细、矩阵导入 |
| 发布检查 | 有 `ProductPublishService.check` | 只查产品身份、模板版本、问题、价格版本、价格规则 | 拆 `PublishCheckEngine`，补工程、BOM、资料、测试用例检查 |
| 发布包 | 有 `pc_product_publish_package` | 快照内容只是轻量 snapshotJson | 发布包要包含配置、工程、价格、BOM、资料版本完整快照 |
| 订单快照 | 有 `ProductOrderSnapshotService` | 只保存发布包、配置选择和输入值 | 改成保存配置快照、价格快照、BOM 快照、资料快照 |
| 导入中心 | 有批次和问题表 | 适合保留，但完整资料包导入后置 | 先保留批次、预览、错误清单 |

## 6. 第一批具体改造顺序

不要先把所有页面铺满。

第一批按这个顺序：

```text
1. 重写 product_capability.sql 的表和菜单基线
2. 新增面料系列 / 面料扩展 / 工程模板 / BOM 规则表
3. 后端新增 Fabric、Engineering、BOM 相关 entity/bo/vo/mapper/service/controller
4. 前端新增面料设置、工程模板、BOM 预览页面
5. 增强 ConfigEvaluationEngine，输出工程和 BOM 所需输入
6. 增强 PriceCalculationEngine，保留现有受控 JSON 匹配
7. 重写 PublishCheckEngine，检查配置、价格、资料、工程、BOM
8. 重写订单快照，保存配置/价格/BOM/资料四类快照
```

第一批完成标准：

```text
一个卷帘或斑马帘产品能完成：
面料维护
销售产品维护
配置问题答案维护
工程模板维护
BOM 预览
价格试算
发布检查
发布包生成
订单快照生成
```

## 7. 当前最大风险

```text
继续补 CRUD，会得到更多页面，但业务仍然不能下单。
继续复用泛 Product Capability 命名，会让销售产品、工程模板、物料、组件边界越来越模糊。
继续只在配置答案里塞 componentJson，BOM 永远无法成为工程规则。
继续只做轻量发布检查，缺资料、缺 BOM、缺工程模板的产品仍可能被发布。
订单快照如果只存 selectedOptions，将来无法复原价格、BOM、资料和生产依据。
```

所以重构第一刀必须从：

```text
数据库表清晰化 + 工程/BOM补齐 + 发布检查真实化
```

开始。
