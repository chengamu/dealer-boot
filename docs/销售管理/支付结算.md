# 支付结算

## 支付事实

每张正式订单创建一个业务 `pay_order`，渠道尝试写入 `pay_order_extension`。支付成功通过幂等回调将订单更新为 `PAID`，金额和币种必须与订单冻结总额一致。

## 支付方式

- PayPal：服务端创建 Order、浏览器审批、服务端 Capture；校验金额、币种和收款方。Webhook 匿名入口必须验签，并在明确支付尝试后跨租户处理；只有“已验签且处理成功/忽略”的事件才按外部事件 ID 幂等返回，`FAILED/PENDING` 允许合法重试。凭据只来自 `PAYPAL_CLIENT_ID`、`PAYPAL_CLIENT_SECRET`、`PAYPAL_WEBHOOK_ID` 等环境变量。
- 银行转账：平台配置收款账户；商家提交付款人、参考号、时间和凭证；平台审核通过或驳回。平台可补录已核实的线下付款。
- 信用额度：额度来自商家档案，账期来自 `credit_term_days`。使用后写信用账户占用、不可变流水和订单应收；逾期、冻结、币种不一致或额度不足时拒绝。

平台审核银行转账、补录付款、登记信用回款、调整或冻结额度时，必须先校验平台身份，再以明确业务主键进入受控跨租户事务；禁止依赖平台租户 `tenant_id = 1` 查询商家记录。

初始化只默认启用不依赖外部密钥的 `credit_limit` 和本地 smoke 使用的 `mock`。PayPal 与银行转账必须完成真实渠道/收款账户配置后启用，初始化脚本不写凭据或虚构银行账户。

## 平台管理

- 支付单列表与详情、支付尝试和 Webhook 事件。
- 银行转账待审核。
- 信用账户、流水和应收回款。
- 线下付款补录、PayPal 对账、支付成功但订单未回写的幂等修复。

核心权限：`pay:order:list/query/submit/supplement/repair`、`pay:bank:submit/review`、`pay:credit:use/query/adjust/freeze/repay`、`pay:reconcile:*`、`pay:channel:*`。

成功支付单不可直接编辑；平台修复必须记录原因和操作人。

## 页面与权限边界

- 商家与内部财务的“付款记录、信用账户、应付账单”使用业务接口，按当前 `tenant_id + business_origin + 部门 data_scope` 限定。
- 平台财务的“支付单据、转账审核、信用账户、信用应收、异常对账”使用 `/platform-finance/**` 和 `platform:finance:*`。
- 平台查询权限不自动包含转账审核、额度调整、账户冻结、信用回款、付款补录、支付修复或忽略异常；这些动作分别使用显式按钮权限。
- 平台商家/门店筛选项由专用只读 options 权限提供，不要求平台财务获得商家资料维护菜单。

## PayPal 外部调用边界

- HTTP 客户端为应用内共享实例，连接超时 3 秒、读取超时 15 秒；创建、Capture、查询、验签和 OAuth 都受超时约束。
- OAuth access token 仅在服务端按 PayPal 返回有效期短期缓存，并预留过期安全窗口；不得写日志或持久化到业务表。
- 创建与 Capture 使用关联当前 `pay_order_extension` 的稳定请求 ID；Capture 从已持久化请求 ID 派生，网络不确定时先查询权威状态。
- 创建 PayPal Order 前，必须在 `REQUIRES_NEW` 事务和订单维度业务锁内先提交 WAITING attempt、`request_id` 及订单关联；外部请求不得包在该事务中。
- 外部调用成功后使用第二个 `REQUIRES_NEW` 事务确认 PayPal order ID 和内部 `PROCESSING` 状态。外部成功但本地超时、事务失败或并发重试时，继续复用已提交 attempt 和同一 `request_id`。
- 日志只记录请求方法、API 路径、稳定请求 ID、耗时和错误类型，不记录 access token、密钥或完整请求/响应报文。
