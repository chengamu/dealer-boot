# 工程测算与报价

## 职责

工程测算用于多房间、多产品的大客户项目；报价管理用于管理测算后形成的客户报价。两者共用 `customer_quote` 底座，但菜单职责不同：

- “工程测算”直接进入测算工作台。
- “报价管理”进入报价列表、客户文件和转订单操作。

## 测算规则

1. 选择启用客户、产品、报价语言和有效期。
2. 录入房间、分数英寸宽高和整数成品数量。
3. 动态配置完全来自正式配方选项，不能写死面料、颜色、控制方式等字段。
4. 父选项变化后清理失效子选项。
5. 产品、尺寸、数量或配置变化后，当前行恢复 `PENDING`。
6. 前端单行测算用于即时反馈；新增或保存草稿时，后端必须权威重算全部明细并在同一事务中落库。
7. 保存接口直接返回事务内最终主档、明细、金额与失败信息，不允许保存后再调用第二次整单计算。
8. `FAIL` 明细允许保存为草稿，但全部行 `PASS` 后才能确认；确认时服务端再次严格测算。

确认冻结产品、正式配方版本、稳定选项和值编码、中英文摘要、BOM、价格命中结果、产品金额及独立邮费快照。英文报价缺少客户可见英文标签时不能确认。

## 转订单

转订单预览读取确认报价快照，并批量匹配商家当前等级和分类/产品类型折扣。前端提交 `expectedTotalAmount`，金额变化时要求重新确认。转换使用 `customer-quote-convert` 锁，成功后写 `source_type=QUOTE`；重复转换幂等返回原订单。

## 接口族

- 草稿：`GET/POST/PUT/DELETE /customer/quotes/**`；`POST/PUT` 保存即权威整单测算并返回完整结果。
- 即时测算：`POST /calculate-item`，只用于单行反馈，不落库。
- 生命周期：`PUT /{id}/confirm`、`PUT /{id}/void`、`POST /{id}/copy`
- 转订单：`POST /{id}/order-preview`、`POST /{id}/convert-order`
- 客户文件：`POST /{id}/export`、`GET /{id}/pdf`、`POST /{id}/email`
- 运行时目录：`/customer/quotes/catalog/products/**`

PDF 字体通过 `PDF_FONT_PATH` 指向可嵌入 CJK TTF/OTF；Linux 推荐 Noto Sans CJK。未配置时使用系统候选字体和兼容回退字体。

## 性能与一致性

- 单次保存或确认创建请求级 `PricingSession`，按租户、可售产品、正式配方版本和币种复用只读价格运行时。
- 同次请求首次加载产品时固定正式配方版本；后续行不得混入新版本。
- 报价运行时只读，禁止调用 `ensureSetting()`、自动初始化或修复价格配置。
- 旧明细删除、真正批量插入新明细、更新主表合计处于同一事务。
- 报价、销售订单和发货分页的统计必须在数据库按 `tenant_id + 业务单据 ID` 分组聚合，只返回计数、数量和完成标志；禁止逐行查询，也禁止为统计加载 BOM、价格或配置快照实体。
- Redis 不承担当前报价正确性；只有真实压测证明数据库读取仍是瓶颈时才评估引入。
