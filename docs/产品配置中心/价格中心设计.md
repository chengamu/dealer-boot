# 价格中心设计草案

生成日期：2026-06-05  
适用范围：SKYSPF / OFBiz 重构中的销售定价、报价解释、价格发布、价格助手。

> 本文只做业务设计，不包含数据库 migration，不直接修改现有老库结构。  
> 配置中心专项设计见：[产品配置中心设计.md](./产品配置中心设计.md)。  
> 配置中心业务可读版见：[配置中心DM.md](./配置中心DM.md)。  
> Agent 体系设计见：[业务Agent体系设计.md](./业务Agent体系设计.md)。

## 0. 当前 base-boot 项目落地口径

价格中心是共享产品能力中心里的销售定价业务域，不是订单系统私有定价表，也不是重做后台基础设施。落到当前项目时：

| 已有能力 | 价格中心处理 |
| --- | --- |
| 权限、菜单、角色 | 新增价格中心菜单和按钮权限，复用现有 Sa-Token 和系统菜单 |
| 前端风格 | 按 `admin-ui` 现有管理端风格实现，不照搬 H5 原型 |
| i18n | 新增可见文案写入 `i18n/locales/en_US.json`，后续同步 |
| 文件导入/附件 | 价格 Excel、说明附件优先复用现有上传能力，价格解析另做业务导入 |
| 审计/UTC | 价格版本、生效时间、审批时间使用现有审计和 UTC 口径 |
| 字典/币种 | 状态、价格类型可走字典；币种优先走现有标准数据 |

价格中心第一版的新增重点是价格方案、价格版本、价格模型、价格测试、价格快照和发布检查；不是新增一套登录、权限、文件、页面框架。

当前可以先和订单系统同仓同库实现，但要按未来独立服务设计：

```text
产品中心 / 价格中心发布价格版本
  -> 报价、订单读取发布版本
  -> 订单保存 PriceSnapshot
  -> ERP / MES / 售后只读价格来源或快照
```

订单系统不能直接拥有或修改正式价格方案。后期拆服务时，价格中心作为 source of truth，订单侧只保存发布版本引用、同步副本和价格快照。

## 1. 核心结论

价格中心应该从配置中心里单独拆出来。

配置中心负责回答：

```text
这个产品是什么？
客户要填哪些问题？
哪些选项可以选？
选项会带出哪些组件和规则？
这个配置能不能卖？
```

价格中心负责回答：

```text
这个配置按哪个销售口径报价？
尺寸、面料、工艺、配件、运费、汇率怎么叠加？
价格来自哪张表、哪个版本、谁审核过？
报价单上每一项金额怎么解释？
```

采购 ERP 负责回答：

```text
真实采购成本是多少？
供应商价格有没有变？
库存、采购、入库、成本核算怎么走？
```

所以第一版不要把“采购成本”“销售报价”“产品配置”塞进同一个页面。配置中心最多显示当前产品引用的价格方案、价格状态和测试入口；真正维护价格，要进入价格中心。

## 2. 为什么必须单独做价格中心

从当前老库和销售 Excel 看，窗帘价格不是一个简单字段。

老系统里有 `PRODUCT_SERIES_PRICE` 这种大矩阵，按产品大类、系列、控制方式、宽高档位取价。它适合卷帘、柔纱帘、蜂巢帘、百叶帘等已有矩阵价。

销售 Excel 里又有另一类逻辑：布帘、罗马帘、穿杆、无拉、电动、韩褶、蛇形帘、柔式、平铺、前穿、后穿、堆叠等工艺，会影响用料米数、加工价、运费和最终美元报价。

面料表里还有 `面料价格（m2）`、`成品价格（m2）` 这类字段，但当前抽样看到 `成品价格` 大多为空，不能直接把它当作完整成品报价。

所以更准确的理解是：

```text
组件类价格：电机、遥控器、太阳能板、充电器、样品、配件包，多数按固定 SKU 价格维护。
窗帘本体价格：由产品系列、面料、尺寸、工艺、用量、加工方式、市场和运费共同决定。
采购成本：由采购/ERP 维护，只能作为复核依据，不能直接替代销售价格。
```

## 3. 新品发布流程

一个新品从配置到上线，建议分四个角色关口：

| 阶段 | 负责人 | 主要工作 | 产物 |
| --- | --- | --- | --- |
| 产品配置 | 设计 / 研发 / 产品运营 | 建产品系列、可售产品、销售变体、问题组、选项、组件绑定、规则 | 配置草稿 |
| 销售定价 | 销售 / 报价负责人 | 维护矩阵价、工艺价、面料销售价、组件加价、运费、汇率口径 | 价格草稿 |
| 采购复核 | 采购 / 成本负责人 | 检查关键物料、供应商成本、替代料、异常毛利 | 复核意见 |
| 审批发布 | 负责人 / 领导 | 检查配置、价格、测试报价、审批记录 | 发布版本 |

发布后，报价和订单只能引用发布版本。订单保存时要同时保存配置快照和价格快照，后面价格改了，老订单不能悄悄变。

## 4. 价格中心的范围

价格中心第一版应该管这些内容：

| 模块 | 用途 | 第一版建议 |
| --- | --- | --- |
| 价格方案 PricePlan | 一个产品或一组产品使用哪套定价逻辑 | 必做 |
| 价格版本 PriceBook | 生效日期、币种、市场、状态、审批 | 必做 |
| 矩阵价格 PriceMatrix | 按宽高档、系列、控制方式取价 | 必做，先可包装老库 |
| 工艺价格 ProcessPricePlan | 布帘/罗马帘等按工艺、倍率、用料计算 | 必做 |
| 面料销售价 FabricSalesPrice | 面料在销售报价中的参考单价 | 必做 |
| 固定组件价 ComponentFixedPrice | 电机、遥控、太阳能板、充电器等 | 必做，但不要过度复杂 |
| 选配加价 OptionSurcharge | 特殊底杆、特殊颜色、服务、包装等 | 必做 |
| 运费方案 ShippingPricePlan | 国内/国外、渠道、实重/抛重、包裹限制 | 必做 |
| 汇率 ExchangeRate | 人民币到美元等报价换算 | 必做 |
| 价格测试 PriceTestCase | 用典型尺寸验证价格命中和明细 | 必做 |
| 折扣 / 客户等级 | 客户价、批发价、活动价 | 后置 |
| 真实采购成本 | 供应商成本、采购价、库存成本 | 价格中心只引用，不维护 |

## 5. 价格模型分类

### 5.1 固定组件价

适合这些东西：

```text
电机
遥控器
太阳能板
充电器
样品
色卡
配件包
```

建议字段：

| 字段 | 含义 |
| --- | --- |
| `componentId` | 组件或标准 SKU |
| `salesPricePolicy` | 是否包含在基础价、单独加价、仅成本复核、单独销售 |
| `unitPrice` | 销售单价 |
| `currency` | 币种 |
| `effectiveFrom/effectiveTo` | 生效日期 |
| `market` | 国内、海外、渠道 |

`salesPricePolicy` 可以先支持：

```text
included_in_base_price     包含在基础价
fixed_addon                固定加价
extra_charge               选中后加价
cost_only                  只做成本复核，不出现在报价
separately_sold            单独作为商品卖
```

### 5.2 矩阵价格

适合老库已有的宽高矩阵：

```text
scope = 产品线 + 系列 + 控制方式 + 面料/结构维度
row = width bracket
column = height bracket
value = sales price
```

第一版不要急着重建 53 万行以上旧价格表，可以先做 `PriceAdapter`：

```text
PriceAdapter:
  source = OLD_PRODUCT_SERIES_PRICE
  input = productCategory + series + controlSystem + width + height
  output = price + matchedRow + matchedColumn + sourceVersion
```

新系统负责把配置结果转成价格查询条件，并保存命中结果。等业务稳定后，再决定是否把旧矩阵归一到新价格表。

### 5.3 面料销售价

从 `卷帘新增面料信息表-价格表.xlsx` 和 `斑马新增面料（含价格上传）.xlsx` 看，面料表里有面料编号、系列、颜色、幅宽、克重、遮光度、面料价格等字段。

这部分不是采购成本表，而是销售报价会参考的面料价格资料。

建议把面料销售价当成一个价格因子：

```text
fabric_sales_amount = fabric_usage * fabric_sales_unit_price
```

但它不一定等于最终窗帘本体价格。对于罗马帘、布帘这类产品，工艺、倍率、用料方式和加工口径会比单纯面料价更重要。

### 5.4 工艺价格方案

这是价格中心最重要的部分。

布帘、罗马帘、穿杆、无拉、电动、有杆、无杆、韩褶、蛇形帘、柔式、平铺、前穿、后穿、堆叠等，不应该靠销售临时翻 Excel 计算。

建议建立 `ProcessPricePlan`：

| 字段 | 含义 |
| --- | --- |
| `processCode` | 工艺编码，如 `FRENCH_PLEAT_2`、`RIPPLE_FOLD`、`ROMAN_CORDLESS` |
| `applicableProductType` | 适用产品类型 |
| `fabricWidthMode` | 定高 / 定宽 |
| `defaultMultiplier` | 默认倍率 |
| `usageFormulaCode` | 用料公式 |
| `laborFactor` | 加工倍率或加工单价 |
| `minChargeRule` | 最小收费规则 |
| `allowedFabricGroup` | 可用面料组 |
| `priceColumnMapping` | Excel/旧表中的价格列映射 |
| `status` | 草稿、待审、已发布、停用 |

布帘报价里抽样看到的思路可以抽象成：

```text
required_usage = usageFormula(width, height, fabric_width, style_multiplier, panel_count)
body_price_rmb = process_unit_price * required_usage * quantity * sales_factor
body_price_usd = body_price_rmb / exchange_rate
final_price = body_price_usd + fixed_addons + option_surcharges + freight_usd
```

这里的 `process_unit_price` 不是单纯采购面料价，而是某个面料在某个工艺下的销售报价口径。

### 5.5 运费方案

罗马帘快递成本表显示，运费会受这些因素影响：

```text
产品宽高
包装加长/加宽/加高
实重
抛重
包裹周长限制
长度限制
国内/国外
快递渠道
```

示例规则：

```text
package_length = width_cm + extra_length_cm
package_width = fixed_width_cm
package_height = fixed_height_cm
volumetric_weight = package_length * package_width * package_height / divisor
billable_weight = max(actual_weight, volumetric_weight)
freight = rateTable(country, channel, billable_weight)
```

如果命中限制：

```text
package_length > 270cm
or 2 * package_width + 2 * package_height + package_length > 300cm
```

价格中心要输出 `BLOCKER` 或转人工复核，不能只给一个错误价格。

## 6. 配置中心和价格中心如何协作

配置中心输出结构化配置：

```text
productId
variantId
selectedOptions
width_cm
height_cm
fabric_code
control_system
process_code
derived_values
component_refs
rule_results
```

价格中心读取这些输入，返回：

```text
pricePlanId
priceBookVersion
currency
priceLines
totalPrice
unmatchedRules
warnings
priceSnapshot
```

配置中心页面只需要显示：

```text
当前产品引用的价格方案
价格方案状态
最近一次价格测试结果
进入价格中心维护/查看的链接
```

不要把完整价格矩阵、工艺价、运费规则全部塞进配置录入页面。

## 7. 价格明细快照

报价和订单必须保存价格快照。

价格快照至少包含：

| 字段 | 用途 |
| --- | --- |
| `pricePlanId` | 命中的价格方案 |
| `priceBookVersion` | 命中的价格版本 |
| `inputSummary` | 宽高、面料、控制方式、工艺、数量 |
| `priceLines` | 每一项金额来源 |
| `shippingLine` | 运费来源和计费重量 |
| `exchangeRate` | 汇率 |
| `totalPrice` | 最终报价 |
| `warnings/blockers` | 超限、缺价、需复核原因 |
| `approvedBy/approvedAt` | 如果人工改价，需要记录 |

价格明细示例：

```text
窗帘本体：ProcessPricePlan ROMAN_CORDLESS_V1，命中 120x180cm，USD 86.30
面料加价：Fabric YZB201，计价面积 2.16 m2，USD 18.36
电机：Bluetooth Motor x 1，USD 32.00
太阳能板：Solar Panel x 1，USD 15.00
运费：US 快递渠道 A，抛重 4.8kg，USD 21.50
合计：USD 175.16
```

## 8. 后台录入体验

价格中心不能让销售每次新品录半小时。第一版应该围绕“快速上线”设计：

正式页面要进入现有管理端菜单体系。下面的“左侧/中间/右侧”是信息架构建议，不是要求按 H5 原型或独立应用重做样式；实现时优先使用当前 `admin-ui` 的表格、抽屉、上传、分页、状态标签和权限按钮。

| 功能 | 目的 |
| --- | --- |
| 从相似产品复制价格方案 | 新品不从空白开始 |
| Excel 批量导入 | 兼容现有销售价格表和面料价格表 |
| 批量应用面料组 | 同一系列几十个颜色不要逐条手录 |
| 工艺模板 | 韩褶、蛇形、罗马帘无拉、电动穿杆等复用 |
| 典型尺寸测试 | 小/中/大/超宽、国内/国外、手拉/电动一键验证 |
| 缺价清单 | 告诉销售哪些尺寸、面料、工艺还没价 |
| 发布前对比 | 新旧价格版本差异、影响产品数量 |

建议页面结构：

```text
左侧：价格方案列表 / 产品引用 / 状态筛选
中间：当前价格方案总览、工艺模板、价格表导入
右侧：测试报价、价格明细、缺价/超限提醒
```

销售只读查看页应该独立存在，用于抄送销售或领导：

```text
价格方案摘要
适用产品
典型尺寸价格
包含/不包含项目
运费口径
生效日期
审批记录
```

## 9. 和 Agent 的关系

`PricingAgent` 可以做：

- 解释价格为什么这么算。
- 根据 Excel 生成价格草稿。
- 发现缺价、重复价、异常毛利、超限报价。
- 帮销售跑典型尺寸测试。
- 输出“还缺什么字段”的追问。

`PricingAgent` 不能做：

- 直接修改正式价格表。
- 绕过销售/采购/领导审批发布价格。
- 把用户一句自然语言当成正式价格依据。
- 回答与业务无关的闲聊。

价格助手的知识来源应该是：

```text
已发布价格方案
配置中心产品数据
销售 Excel 抽取结果
业务设计文档
用户纠错后沉淀的结构化规则
```

第一版不一定需要向量数据库。资料规模可控时，结构化表 + 全文检索 + 规则节点，比把所有内容都塞给大模型更稳。

## 10. 第一版落地建议

第一版建议只做这些：

1. 价格中心独立入口和只读总览。
2. `PricePlan / PriceBook / PriceSnapshot` 概念。
3. 固定组件价。
4. 老库矩阵价 `PriceAdapter`。
5. 工艺价格方案 `ProcessPricePlan`。
6. 面料销售价导入。
7. 运费方案导入和测试。
8. 典型尺寸价格测试。
9. 缺价 / 超限 / 未审批拦截。

先不要做：

- 客户等级复杂折扣。
- 促销活动。
- 全量成本核算。
- 自动采购成本联动。
- Agent 自动发布正式价格。

这样配置中心能保持稳定，价格中心能服务销售真实报价，ERP 后期也能接入采购成本和审批流程。
