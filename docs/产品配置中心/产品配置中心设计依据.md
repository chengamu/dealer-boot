# 产品配置中心设计依据

生成日期：2026-06-04  
关联文档：

- [配置中心DM.md](./配置中心DM.md)
- [产品配置中心设计.md](./产品配置中心设计.md)

本文说明：为什么要做“产品配置中心”，以及现有设计依据来自哪里。

本文不是数据库建表说明，也不是开发任务清单。它用于说明设计判断的来源：业务交流、参考系统观察、老系统数据库抽样、现有 Excel 产品资料，以及当前人工录入方式。

补充当前项目背景：`base-boot` 已经不是空白系统。登录权限、系统菜单、字典、OSS 文件、i18n、审计、租户、前端管理端风格和后端通用 CRUD 结构已经存在。本文论证的是为什么还需要新增“共享产品能力中心”这个业务基础层，不是论证要重建一套后台基础平台。后续落地应复用现有基础能力，只新增产品基础信息、配置、价格、组件/BOM、资料绑定、发布检查和快照等业务域能力。

这套能力当前可以先寄居在订单系统里，方便下单和报价联调；但它不应该长期被建模成订单系统私有数据。ERP、MES、报价、售后和客户门户也会消费这些发布后的产品能力。

## 1. 设计结论从哪里来

这次设计不是凭空画出来的，主要来自五类依据：

| 依据来源 | 看到的问题 | 推导出的设计 |
| --- | --- | --- |
| 业务交流 | 新产品现在靠 Excel、字段说明、人工录入，规则容易漏 | 需要统一的配置中心和导入草稿 |
| HD/HB 截图参考 | 下单是按产品配置一步步选择，系统动态显示、校验、算价 | 需要“下单模板 + 规则 + 价格 + 配件” |
| 现有数据库只读抽样 | 产品、分类、颜色、面料、配件、价格表混在一起 | 需要拆清产品、选项、组件、价格规则 |
| 现有 Excel 资料 | 百叶帘价格按材料、宽高面积、取整规则计算 | 需要公式价格、材料价格、尺寸规则 |
| 现行业务方式 | 电动自动带电机，手拉自动带绳子/链条，超宽要特殊方案 | 需要规则生成 BOM/配件，而不是人工备注 |

核心结论：

```text
配置中心不是后台多做几个管理页面。
配置中心是报价、订单、生产读取产品数据的统一入口。
```

## 2. 我们这次交流提炼出的业务判断

### 2.1 配置中心是基础，不是订单的附属功能

原来的理解容易从“订单怎么下”开始。

但这次交流后，结论变成：

```text
先有产品配置中心
再有报价
再有订单
再有生产
```

原因很直接：

```text
客户下单选什么，由产品配置决定。
价格怎么算，由产品配置决定。
自动带什么配件，由产品配置决定。
生产需要什么 BOM，也由产品配置决定。
```

如果配置中心不清楚，报价、订单、生产都会各自补规则，最后系统会越来越乱。

### 2.2 现在痛点不是“没有产品”，而是产品规则没有标准入口

现在新增产品通常会拿到：

```text
产品 Excel
价格表
字段说明
颜色/面料/材料资料
配件资料
图片
一些口头规则或备注
```

然后人工录入系统。

问题是：

```text
价格表在哪里维护？
材料和颜色关系在哪里维护？
电动和手拉对应的配件在哪里维护？
尺寸限制在哪里维护？
超宽、超高、特殊方案在哪里维护？
客户下单页面从哪里读这些规则？
```

所以需要一个统一的配置中心，把这些东西变成标准数据。

## 3. HD/HB 参考系统带来的启发

这里的 HD/HB 指我们讨论中的 Hunter Douglas 参考系统。以下依据来自你提供的视频截图和我们对下单流程的观察，不等同于官方技术实现说明。

### 3.1 它不是简单的“产品录入表”

从截图和流程看，参考系统更像一个配置器：

```text
选择产品或产品系列
加载该产品可用的配置项
选择控制方式、面料、颜色、安装方式、尺寸等
系统动态显示可选项
系统提示不合法组合或尺寸问题
系统计算价格
系统生成订单配置
```

这说明产品后台不是只存一个 SKU。

它必须知道：

```text
这个产品要问客户哪些问题
每个问题有哪些答案
答案之间有什么限制
选完后价格怎么算
选完后需要哪些配件
```

### 3.2 下单体验说明后端需要规则

如果客户选择“电动”，页面应该出现电机、电源、遥控、太阳能板等相关选项。

如果客户选择“手拉”，页面应该出现链条、拉绳、控制方向等相关选项。

如果某个面料不支持某种尺寸或某种控制方式，页面应该直接拦截或提示。

这背后的后端不是简单字段，而是规则：

```text
显示隐藏规则
必填规则
可选/不可选规则
尺寸限制规则
价格规则
配件生成规则
```

### 3.3 参考系统给我们的设计方向

| 参考系统表现 | 我们应该沉淀的能力 |
| --- | --- |
| 下单时按步骤选择 | 配置模板 |
| 不同产品显示不同字段 | 配置项和适用范围 |
| 切换控制方式后选项变化 | 显示隐藏规则 |
| 不支持的组合不能选 | 兼容规则 |
| 尺寸超限会提示 | 尺寸规则 |
| 价格自动计算 | 价格规则 |
| 选完后形成订单 | 配置快照 |

## 4. 老系统数据库抽样带来的依据

老系统只读抽样说明，它已经有配置器雏形，但概念没有拆清楚。

抽样表和行数：

| 表 | 行数 | 说明 |
| --- | ---: | --- |
| `PRODUCT` | 952 | 产品、组件、颜色、面料等混用 |
| `PRODUCT_CATEGORY` | 166 | 产品大类、系列、控制方式、外壳类型等混用 |
| `PRODUCT_CONFIG_ITEM` | 48 | 类似配置项，也就是下单时要选的问题 |
| `PRODUCT_CONFIG_OPTION` | 601 | 类似配置项选项 |
| `PRODUCT_CONFIG_PRODUCT` | 519 | 选项映射到产品，类似 BOM/配件映射 |
| `PRODUCT_CONFIG_CONFIG` | 7276 | 已保存配置组合 |
| `PRODUCT_SERIES_PRICE` | 530631 | 按产品大类、系列、控制方式、宽高维护价格矩阵 |

这说明几个事实：

```text
老系统不是完全没有配置逻辑。
老系统已经在维护配置项、选项、价格矩阵、选项到产品的映射。
但这些概念混在老 OFBiz 结构里，不够清晰，也不方便扩展。
```

### 4.1 现有产品录入方式的特点

从老库可以看到：

```text
一个 PRODUCT 可能是销售产品
一个 PRODUCT 也可能是颜色
一个 PRODUCT 也可能是面料
一个 PRODUCT 也可能是电机、太阳能板、配件
```

这样做短期能跑，但后期会有问题：

```text
销售产品和生产组件边界不清
颜色/面料和真实商品边界不清
配置选项和 BOM 组件边界不清
导入新产品时不知道应该录到哪里
下单结果难以结构化输出
```

所以新设计要把它们拆成业务概念：

| 老系统混合概念 | 新配置中心概念 |
| --- | --- |
| 销售产品 | 可售产品 |
| 卷帘、百叶帘等大类 | 产品模型 |
| 客户下单要选的问题 | 配置项 |
| 客户可以选择的答案 | 选项 |
| 电机、支架、底杆、太阳能板 | 组件/配件 |
| 选项映射产品 | 配件/BOM 规则 |
| 宽高价格表 | 价格规则/价格矩阵 |

### 4.2 老系统下单和计价方式的依据

老库中的 `PRODUCT_SERIES_PRICE` 数量很大，说明老系统已经大量依赖尺寸价格矩阵。

可以理解为：

```text
产品大类 + 系列 + 控制方式 + 宽度档 + 高度档 = 价格
```

这类价格不能简单写成一个固定价格字段。

新配置中心必须支持：

```text
宽高矩阵价格
价格版本
生效日期
产品/系列/控制方式作用范围
价格测试
```

## 5. Excel 产品资料带来的依据

你提供的百叶帘资料说明，某些产品不是按矩阵价，而是按公式价。

百叶帘资料里的价格逻辑可以概括为：

```text
宽厘米 = 宽英寸 × 2.54
高厘米 = 高英寸 × 2.54
计价面积 = 向上取整(宽厘米 × 高厘米 / 10000)
价格 = 计价面积 × 材料平方价
```

材料平方价示例：

| 材料 | 平方价 |
| --- | ---: |
| 椴木 | 22.01 |
| 楠竹 | 22.01 |
| 桐木 | 20.95 |
| PVC-PS | 16.72 |

尺寸范围示例：

| 项目 | 范围 |
| --- | --- |
| 宽度 | 12 到 118 inch |
| 高度 | 12 到 160 inch |

颜色资料也说明：

```text
不同材料对应不同颜色组。
WPS 有自己的颜色。
椴木有自己的颜色。
桐木有自己的颜色。
竹有自己的颜色。
```

所以新配置中心必须支持：

```text
材料控制颜色组
材料控制平方价
尺寸范围校验
按面积取整计价
Excel 导入后进入草稿
人工审核后发布
```

## 6. 为什么价格要拆成多种规则

窗帘产品的价格不是单一价格。

实际报价往往由这些部分组成：

```text
基础价格
+ 面料价格
+ 铝材/轨道/外壳/底杆价格
+ 拉杆/拉绳/链条价格
+ 电机/遥控器/太阳能板价格
+ 特殊颜色或特殊工艺加价
+ 安装、包装、服务等费用
= 最终报价
```

因此价格中心不能只有：

```text
product_id -> price
```

至少要支持这些价格类型：

| 价格类型 | 适用产品 | 例子 |
| --- | --- | --- |
| 固定价格 | 标准品、配件、样品 | 遥控器、太阳能板、充电器 |
| 矩阵价格 | 老系统已有价格表 | 宽度档 × 高度档 |
| 面积价格 | 百叶帘、部分定制品 | 计价面积 × 材料平方价 |
| 面料价格 | 卷帘、柔纱帘、蜂巢帘 | 面料用量 × 面料单价 |
| 铝材价格 | 轨道、外壳、底杆、型材 | 长度 × 铝材单价，或按档位查价 |
| 拉杆/拉绳/链条价格 | 手拉产品 | 固定价、长度价、颜色/规格加价 |
| 电机配件价格 | 电动产品 | 电机、遥控器、太阳能板 |
| 选项加价 | 特殊颜色、特殊底杆、特殊安装 | 选中后加价 |

### 6.1 卷帘价格例子

```text
客户选择：L盒卷帘
控制方式：电动
面料：YS180002
尺寸：宽 120 inch，高 90 inch

系统计算：
  面料价格 = 面料计价面积 × YS180002 单价
  铝材价格 = 外壳/底杆长度 × 对应铝材单价
  电机价格 = Bluetooth Motor × 1
  支架价格 = 支架规则带出
  太阳能板价格 = 如果选择 Yes，则 Solar Panel × 1
  特殊加价 = 特殊底杆/特殊颜色/特殊安装
```

如果宽度超过阈值：

```text
如果 宽度 > 157 inch
  电机数量 = 2
  电机价格 = Bluetooth Motor × 2
  提示：该宽度需要双电机方案
```

### 6.2 手拉产品价格例子

```text
客户选择：手拉卷帘

系统计算：
  面料价格
  铝材价格
  链条/拉绳价格
  底杆和支架价格
```

手拉不应该带电机，电动不应该带链条/拉绳。

这不是价格问题，而是规则和配件生成问题。

## 7. 为什么 BOM/配件要由规则生成

你提到的业务例子很典型：

```text
电动有电机。
手拉有绳子/链条。
有些配件只有某些产品或某些控制方式支持。
长度超过 4 米可能要两个电机。
```

这说明配件不能只靠人工勾选。

应该由配置结果自动生成：

```text
客户选择电动
  -> 自动带电机

客户选择手拉
  -> 自动带链条/拉绳

客户选择太阳能板
  -> 自动带 Solar Panel

宽度超过阈值
  -> 自动带两个电机或提示双电机方案
```

如果不这样做，会有几个问题：

```text
销售容易漏配件。
生产拿不到稳定组件清单。
报价和生产口径不一致。
新产品导入后仍然靠人工经验。
```

## 8. 为什么标准产品也要进配置中心

配置中心不只管定制窗帘。

标准产品也应该进入配置中心：

```text
遥控器
太阳能板
充电器
色卡
样品
固定规格成品
配件包
```

原因是：

```text
标准品可以单独卖。
标准品也可能被定制产品自动带出。
标准品也需要图片、说明、价格、编码、状态。
订单中也要保存当时卖的是哪个版本、哪个价格。
```

所以配置中心要支持三种销售模式：

| 销售模式 | 说明 | 例子 |
| --- | --- | --- |
| 标准品 | 不需要复杂配置 | 遥控器、太阳能板 |
| 半标准品 | 固定 SKU，但有少量选项 | 色卡选择颜色、样品选择规格 |
| 定制品 | 需要尺寸、规则、价格、配件 | 卷帘、百叶帘、蜂巢帘 |

## 9. 为什么要保存配置快照

订单不能只保存一句描述。

比如不能只保存：

```text
L Cassette Motor Solar Shades, white, inside mount
```

必须保存结构化结果：

```text
产品是什么
用的是哪个配置模板版本
客户选了哪些选项
宽高是多少
当时价格怎么算出来
当时自动带了哪些配件
当时哪些规则通过或拦截
```

原因：

```text
产品以后会改。
价格以后会改。
颜色以后会停用。
规则以后会调整。
老订单必须能复原。
生产和售后必须知道当时到底怎么选的。
```

所以 `ConfigurationSnapshot` 是必须项，不是可选项。

## 10. 这份设计如何回应当前系统问题

| 当前问题 | 新设计怎么解决 |
| --- | --- |
| 产品、颜色、面料、组件混在 PRODUCT | 拆成产品、选项、组件 |
| 分类里混了产品线、系列、控制方式 | 用产品模型、系列、配置项区分 |
| 价格表巨大但缺规则版本 | 价格中心管理矩阵/公式/版本 |
| 配件带出逻辑不清 | 用 BOM/配件规则生成 |
| 新产品靠 Excel 人工录 | 导入中心先生成草稿，再人工审核 |
| 前端下单字段不统一 | 从配置模板读取字段 |
| 客户切换方式后选项支持变化 | 用显示隐藏和兼容规则 |
| 老订单难复原 | 保存配置快照 |

## 11. 第一版为什么先做人工配置中心

你也明确说过，Agent 可以后期做。

这个判断是对的。

原因：

```text
如果没有标准配置中心，Agent 也不知道要把资料整理成什么结构。
如果后台不能人工设置清楚，自动导入只会把混乱放大。
如果没有测试和发布流程，Agent 生成的配置无法安全上线。
```

第一版应该先做：

```text
产品基础信息
配置项
选项
规则
价格
配件
测试发布
订单快照
```

等这些结构稳定后，再做：

```text
Excel/PDF/图片资料导入
Agent 自动整理草稿
人工审核差异
发布版本
```

## 12. 本设计的边界

本文支持当前方向：

```text
统一产品配置中心
支持定制品和标准品
支持规则下单
支持价格拆分
支持配件自动带出
支持配置快照
支持以后导入和 Agent
```

本文不直接做：

```text
修改现有数据库结构
写 migration
强行迁移所有老产品
一次性对接所有后期系统
让 Agent 自动发布配置
```

## 13. 总结

这套设计有据可依：

```text
业务上：新产品资料和规则需要统一录入。
体验上：参考系统证明客户下单需要动态配置器。
数据上：老系统已有配置项、选项、价格矩阵、配件映射雏形。
资料上：Excel 证明价格既有矩阵，也有面积公式和材料价。
生产上：电机、链条、铝材、拉杆等组件必须由规则带出。
```

所以配置中心的核心不是“多建几张表”，而是把以前散落的产品、价格、规则、配件、说明统一成一个可发布、可测试、可下单、可追溯的产品基础层。
