# 共享产品能力中心开发入口

生成日期：2026-06-07

本文是共享产品能力中心的开发入口文档。开发、联调、验收先看本文；其他文档作为专题详设和背景资料引用，不再各自维护一套重复的开发计划。

## 1. 先读结论

- 这是共享产品能力中心，不是订单系统的私有配置页；当前可以寄居订单系统，模块和数据库边界必须按后续可拆服务设计。
- 正式菜单是三个一级菜单：`基础资料`、`配置与价格`、`发布与应用`。其中 `基础资料` 就是产品配置域的“基础信息设置”，不再使用一个笼统的“产品中心/产品能力”一级菜单承载全部功能。
- 普通维护页走标准 grid/list + 右侧抽屉；只有工作台、配置录入工作台、价格编辑与测试、测试发布、销售只读总览允许自定义内容区。
- 权威数据来自后端：配置求值、价格计算、BOM、发布检查、快照、审核结果不能由前端临时推导。
- 发布包、订单快照、价格快照、BOM 快照、资料版本等权威数据不放 Redis；Redis 最多用于短期查询缓存或会话类能力。
- 新增前端页面必须接入现有 `admin-ui` Layout、动态路由、菜单权限、i18n、UTC、请求封装和 Element Plus/SvgIcon 体系。
- 页面 SFC 只做编排；复杂区块优先复用公共组件，没有再沉淀新组件，禁止一个 Vue 文件写到底。

## 3. 开发顺序

第一批不再按“先铺骨架就算完成”推进，而是按可验收闭环推进。

| 顺序 | 工作包 | 交付内容 | 验收口径 |
| --- | --- | --- | --- |
| 0 | 历史界面和生成器标准化 | 代码生成器 Vue 模板改为抽屉基线；系统用户、商家用户、菜单、角色、参数、字典、岗位、部门、OSS、公告、操作日志详情等历史页收敛到新标准 | 新增/编辑/详情默认右侧抽屉；短确认仍用弹窗；这些页面成为普通 grid 标准样例 |
| 1 | 数据库和菜单基线 | 执行受控 SQL；落三个一级菜单和所有二级菜单；补按钮权限 | 刷新页面不 404；菜单/按钮/后端权限三方一致；真实库可查 |
| 2 | 后端模块和基础 CRUD | `bocoo-modules-product` 或明确的独立产品能力模块；分类、物料、组件、资料、模型、变体、问题组、价格方案、缺口、发布包等基础接口 | Controller/Service/Mapper/Entity/BO/VO 风格符合生成器；导出、状态、引用检查权限可控 |
| 3 | 标准 grid/list 页面 | 产品分类、物料管理、辅材管理、资料资产、资料绑定、产品模型、销售变体、问题组模板、审核审批、缺口待办、发布包、同步日志、导入中心 | 搜索、重置、分页、toolbar、right-toolbar、抽屉、权限、i18n、UTC 都通过浏览器验证 |
| 4 | 5 个主自定义页 | 工作台、配置录入工作台、价格编辑与测试、测试发布、销售只读总览 | 以设计文档、H5 静态参考和 PNG 效果图为输入，转为当前 `admin-ui` 内的 Vue 组件化页面；底部可见；关键操作不绕路；复杂区块组件化 |
| 5 | 业务引擎闭环 | 配置求值、价格计算、BOM 预览、发布检查、快照实例、read model、outbox | 前端只展示后端结果；发布包不可变；订单/ERP/MES 可读取冗余快照 |
| 6 | 报价、审核、发布闭环 | 报价预览、Quick Quote、审核通过/拒绝、缺口转任务、发布包详情、同步重试 | 不再停留在隐藏 tab 或 TODO；真实角色授权后可完成流程 |
| 7 | 浏览器真实验收 | 用 `192.*` 实际地址、真实登录态、真实开发库跑完整流程 | 菜单、刷新、接口、权限、视觉、录入便捷性、前后端流程全部过一遍 |

## 5. 类型字典口径

物料管理和辅材管理里的“类型”先走现有 `sys_dict`，不在 P0 新建独立“类型管理模块”。

原因：

- 面料、电机、遥控器、下杆、辅材、服务项这类是筛选、下拉、导入校验和展示翻译用的稳定枚举。
- 当前项目按字典类型取数据的接口 `/system/dict/data/type/{dictType}` 对非公开字典会检查登录态，但不做菜单按钮权限过滤；登录后页面下拉可以正常读取。
- 字典管理本身仍归系统管理权限控制，普通业务用户只读使用字典，不维护字典。
- 真正的业务主数据仍在 `pc_material` 和 `pc_component`：名称、编码、属性、供应商、适用范围、价格引用、BOM 引用都不能放到字典里。
- 类型下拉可以走字典，但不是所有“类型能力”都只做字典；一旦涉及属性模板、字段校验、适用范围、默认规则、发布检查、价格和 BOM 行为，必须落到物料/辅材业务表和 Service/Engine 中。
- 所有类型下拉必须支持模糊过滤；物料、辅材、产品模型、销售变体、问题组、价格方案等对象选择必须支持远程搜索或选择器，不允许只给长列表下拉。

P0 必备字典建议：

| 字典类型 | 用途 | 建议值 |
| --- | --- | --- |
| `product_material_type` | `pc_material.material_type` 物料类型 | `FABRIC` 面料、`COLOR` 颜色/色卡、`MOTOR` 电机、`REMOTE` 遥控器、`HUB` 网关、`SOLAR_PANEL` 太阳能板、`PROFILE` 型材、`BOTTOM_RAIL` 下杆、`ACCESSORY` 辅材/配件、`PACKAGING` 包装件、`SERVICE` 服务项、`FINISHED_GOOD` 标准成品 |
| `product_component_type` | `pc_component.component_type` 辅材/组件类型 | `MOTOR` 电机、`REMOTE` 遥控器、`HUB` 网关、`SOLAR_PANEL` 太阳能板、`BOTTOM_RAIL` 下杆、`BRACKET` 支架、`CHAIN` 链条、`TUBE` 卷管、`PROFILE` 型材、`FABRIC` 面料、`ACCESSORY` 配件、`PACKAGING` 包装件、`SAMPLE` 样品 |
| `product_business_type` | `pc_material.business_type` 后期物料口径 | `FINISHED` 成品、`CUSTOM` 定制品、`RAW` 原料、`PURCHASE` 采购件、`SERVICE` 服务 |
| `product_unit` | 单位下拉和导入校验 | `cm`、`inch`、`m`、`m2`、`PC`、`Set` |

如果后续某个类型需要字段模板、属性 schema、默认规则或权限策略，再在物料/辅材业务模块里扩展属性配置；不要为了类型下拉提前新增一套类型表。

## 6. Done 口径

`.ai/TASKS.md` 里此前出现“Batch 1 到 Batch 10 的 done 只表示骨架/MVP/草案通过编译或空库验证，不表示真实业务体验完成”，根因不是某一份设计稿写错，而是计划体系把不同层级的完成状态混在了一个 `done` 字段里。

以后按下面口径判断：

| 状态 | 可以说明什么 | 不能说明什么 |
| --- | --- | --- |
| 草案完成 | 文档、表结构、API 分组或 UI 方向已形成 | 不能说明已可开发完或可上线 |
| 骨架完成 | 模块、路由、菜单、接口或页面能编译/启动 | 不能说明业务流程可用 |
| 空库验证完成 | SQL 可在临时库执行，基础约束无语法错误 | 不能说明真实库、真实菜单、真实数据、真实权限没问题 |
| 编译通过 | 代码没有明显编译错误 | 不能说明接口联通、页面好用、权限正确 |
| 业务闭环完成 | 用户可按真实流程完成新增、编辑、求值、试算、审核、发布、查看 | 才能进入体验验收 |
| 体验验收完成 | 浏览器真实验证视觉、操作、滚动、底部按钮、错误提示、刷新、权限、i18n、UTC | 才能称为该页面/功能完成 |

任务管理里如果只是骨架、草案、空库或编译通过，不能再直接写 `done` 并让人误解为功能完成；必须在任务标题或验收备注里写清楚 `骨架完成`、`草案完成`、`空库通过`。真正的 `done` 必须满足本文第 7 节的验收门。

## 7. 每个工作包的验收门

每个工作包完成前至少过下面这些门：

| 验收门 | 要求 |
| --- | --- |
| 菜单门 | 数据库 `sys_menu` 有正式菜单和按钮；刷新 URL 不 404；不靠临时 alias 长期兜底 |
| 权限门 | Vue `v-hasPermi`、Java `@SaCheckPermission`、`sys_menu.perms` 三方一致 |
| API 门 | 页面所有按钮有真实接口或明确禁用态；不能只前端假按钮 |
| 数据门 | 读写真实开发库；列表、详情、导出、状态、引用检查有真实字段 |
| UI 门 | 普通页符合标准 grid/list；复杂页完成设计参考到 Vue 组件化页面的重构优化；底部操作可见 |
| 组件门 | 复杂区块组件化，优先复用公共组件；页面 SFC 不写到底 |
| 选择器门 | 字典下拉支持 `filterable`；业务对象选择支持模糊过滤、远程搜索或选择器；长列表不允许只靠普通下拉 |
| i18n 门 | 新增可见文案只写 `i18n/locales/en_US.json`，不手写 public runtime JSON |
| UTC 门 | 后端 UTC 语义，前端使用 `formatUtc()` / `toUtcPayload()` 等统一工具 |
| 快照门 | 发布包、配置快照、价格快照、BOM 快照、资料版本可追溯，不依赖 Redis 权威缓存 |
| 浏览器门 | 用真实地址和真实登录态完成菜单进入、刷新、搜索、录入、保存、错误提示和关键流程 |

## 8. 当前 P0 缺口

以下内容不能因为已有骨架或 TODO 就从计划里消失：

| 缺口 | 处理要求 |
| --- | --- |
| 配置模板工作台视觉和交互 | 按新版 H5 主参考继续还原，去掉工程调试感，保留检查、预览、BOM 同屏反馈 |
| 价格中心 | 价格矩阵、右侧试算器、命中明细、导入预览必须形成操作闭环 |
| 报价预览 / Quick Quote | 必须有独立二级入口，并从销售只读页可进入 |
| 审核审批 | 审核通过、拒绝、原因、日志、按钮权限和后端权限必须完整 |
| 缺口待办 | 发布检查 BLOCKER/WARNING 能转缺口并定位修复入口 |
| 发布包 | 详情、版本、同步状态、不可变快照、下载/查看权限必须清晰 |
| 同步日志 / Outbox | ERP/MES/订单同步状态、失败原因、重试入口必须保留 |
| 导入中心 | XLS/XLSX 解析预览只是基础，后续还要字段映射、冲突处理、提交入库 |
| 历史页面标准化 | 旧弹窗长表单不豁免，必须先收敛为新标准样例 |

## 9. 开发禁止项

- 不把高频主功能长期藏在一个页面 tab 里。
- 不新增侧边栏三级菜单。
- 不把 Entity 直接返回给前端。
- 不在 Controller 或前端复制配置、价格、BOM、发布检查规则。
- 不让订单、ERP、MES 直接写共享产品能力核心表。
- 不把 H5 静态页、Tailwind CDN、外链图片、FontAwesome、写死数据直接搬进生产。
- 不用 Redis 保存发布包、审核结果、订单快照、价格快照、BOM 快照的权威数据。
- 不新增 MySQL SQL，不修改生产环境配置。

## 10. 开发执行建议

每次领取任务时按这个顺序做：

1. 从本文确认菜单、页面类型、类型字典和验收门。
2. 到专题文档读取该工作包的数据库、API、UI 细节。
3. 先补数据库菜单和按钮权限，再做前后端实现。
4. 普通页面先按生成器标准做 grid/list + 抽屉；复杂页面先拆 Vue 组件，再把设计文档、H5 静态参考和 PNG 效果图转成可维护的项目内实现。
5. 后端先保证 Service/Engine 权威结果，再让前端展示。
6. 自测时必须用真实浏览器、真实开发库、真实登录态验证。
7. 只有通过第 7 节验收门，任务才允许标记为功能完成。

一句话：本文负责告诉开发“从哪里开始、按什么顺序做、什么才算完成”；专题文档只负责解释“为什么这样设计”和“细节字段怎么落”。
