> 后续真正落库前，需要再拆成技术实施计划、表级评审和 PostgreSQL migration。  
> 当前项目 PostgreSQL 是主数据库；新设计不补 MySQL SQL。

## 0. 核心结论

共享产品能力中心不是订单系统私有数据。它维护产品基础、配置、组件、资料、价格、规则、BOM、发布包和快照来源，订单、报价、ERP、MES、售后只消费发布后的能力。

数据库设计采用四层：

```text
编辑源数据层
  -> 维护产品、物料、组件、配置问题、规则、价格草稿
  -> 适度规范化，方便维护和复用

发布包层
  -> 发布时生成不可变宽表 + JSONB
  -> 固化产品名、分类名、问题答案、规则摘要、BOM、资料、价格引用

订单快照层
  -> 下单/报价时生成完全自包含快照
  -> 历史订单不依赖当前产品表也能复原

读模型 / 同步事件层
  -> 给列表、搜索、AI、工作台、ERP/MES 同步读取
  -> 避免每次查询 join 十几张表
```

关键原则：

| 原则 | 说明 |
| --- | --- |
| 平台功能优先 | 产品能力中心按平台共享主数据设计，不按商户租户拆多份配置 |
| 发布后不可变 | 草稿可以改，发布版本不能原地改；变更必须生成新版本 |
| 快照自包含 | 快照保存当时的 code、name、version、单位、币种、计算明细，不只存 ID |
| 宽表用于读 | 发布包、快照、read model 强冗余，换查询性能和可追溯性 |
| 源数据适度规范 | 编辑态不要全部塞宽表，否则维护和复用会失控 |
| 全字段中文注释 | 后续每张表、每个字段都必须有中文注释，遵循现有 `COMMENT ON TABLE/COLUMN` 规范 |
| UTC 语义 | 时间字段使用 `timestamptz`，字段注释明确 `UTC timestamptz` |
| 不强依赖跨系统外键 | 未来可能独立服务，订单/ERP/MES 不应强 FK 到产品中心核心表 |

## 1. 当前项目数据库规范

根据当前项目已有 PostgreSQL 初始化脚本和基础类，后续落库应遵守：

| 规范 | 当前项目口径 | 本设计处理 |
| --- | --- | --- |
| 主键 | `bigint PRIMARY KEY`，由 MyBatis-Plus 雪花 ID 生成 | 所有业务主键使用 `bigint` |
| 时间 | PostgreSQL `timestamptz`，连接会话 UTC | 所有时间字段注释写明 `UTC timestamptz` |
| 审计字段 | `create_by_id/create_by/create_time/update_by/update_time` | 编辑态表使用完整审计字段 |
| 软删除 | `del_flag char(1) DEFAULT '0'`，`0存在，2删除` | 编辑态核心表保留软删除；发布包不物理删除 |
| 状态 | 多数用 `status char(1)` 或业务状态字符串 | 简单启停用 `char(1)`；流程状态用 `varchar(20)` |
| 租户拦截 | 拦截字段名是 `tenant_id`，忽略表走 `bocoo.tenant.ignore-tables` | 产品中心按平台功能设计，落库时要明确是否加入忽略表 |
| 注释 | 使用 `COMMENT ON TABLE`、`COMMENT ON COLUMN` 中文注释 | 后续所有表和所有字段必须写中文注释 |
| JSON | 项目使用 PostgreSQL，可使用 `jsonb` | 发布包、快照、规则内容、读模型使用 `jsonb` |

### 1.1 平台功能和租户处理

用户当前判断是：这块都是平台功能，租户可以不作为业务维度。这个判断合理。

但当前项目启用了租户拦截器，所以技术落地不能完全不处理租户。推荐二选一：

| 方案 | 说明 | 建议 |
| --- | --- | --- |
| 方案 A：加入租户忽略表 | 产品中心核心表加入 `bocoo.tenant.ignore-tables`，业务上不带 `tenant_id` 查询 | 推荐用于平台共享主数据 |
| 方案 B：统一平台租户 | 表保留 `tenant_id bigint NOT NULL CHECK (tenant_id <> 0)`，所有平台产品能力写 `tenant_id = 1` | 适合需要复用现有 BaseEntity 和租户上下文的阶段 |

第一期建议：

- 核心产品能力表按平台共享主数据看待。
- 不做商户级配置复制。
- 价格如果后期存在商户专属价，再通过 `scope_type` 扩展，不影响当前平台主价。
- 无论选 A 还是 B，都要在技术实施计划里逐表确认，不要让租户拦截器误伤平台数据查询。

## 2. 命名和字段约定

### 2.1 表名前缀

建议使用 `pc_` 作为共享产品能力中心表前缀，含义是 Product Capability。

| 前缀 | 用途 | 示例 |
| --- | --- | --- |
| `pc_` | 产品能力中心核心业务表 | `pc_product_model` |
| `pc_published_` | 发布后不可变表 | `pc_published_package` |
| `pc_read_` | 宽读模型表 | `pc_read_model` |
| `pc_event_` | 同步事件和 outbox 表 | `pc_event_outbox` |

索引命名遵循当前项目习惯：

| 类型 | 命名 |
| --- | --- |
| 普通索引 | `idx_表名_字段简写` |
| 唯一索引 | `uk_表名_字段简写` |
| 部分唯一索引 | 使用 `WHERE del_flag = '0'` 或业务条件 |

### 2.2 通用字段

后续落 SQL 时，所有字段都要有中文注释。下面是通用字段注释标准：

| 字段 | 类型建议 | 中文注释 |

| `update_time` | `timestamptz` | 更新时间，UTC timestamptz |

| `last_error` | `varchar(1000)` | 最近一次错误信息 |
| `created_time` | `timestamptz` | 事件创建时间，UTC timestamptz |
| `processed_time` | `timestamptz` | 事件处理时间，UTC timestamptz |
| `create_by_id` | `bigint` | 创建者ID |
| `create_by` | `varchar(64)` | 创建者 |
| `create_time` | `timestamptz` | 创建时间，UTC timestamptz |
| `update_by` | `varchar(64)` | 更新者 |
| `update_time` | `timestamptz` | 更新时间，UTC timestamptz |

建议索引：

- `uk_pc_event_outbox_no(event_no)`
- `idx_pc_event_outbox_sync(sync_status, next_retry_time, created_time)`
- `idx_pc_event_outbox_aggregate(aggregate_type, aggregate_id, aggregate_version_no)`

## 8. JSONB 设计规则

JSONB 不是偷懒字段。只有以下场景适合用：

| 场景 | 是否适合 JSONB | 原因 |
| --- | --- | --- |
| 发布包完整结构 | 适合 | 发布后不可变，读取多于编辑 |
| 订单快照 | 适合 | 必须自包含，历史复原优先 |
| 规则条件和动作 | 适合 | 结构多变，第一期不宜拆太细 |
| 价格矩阵和公式 | 适合 | 价格类型复杂，第一期用统一规则项承载 |
| 列表筛选字段 | 不适合只放 JSONB | 常查字段必须拉成普通列 |
| 状态和编码 | 不适合只放 JSONB | 必须做索引和权限判断 |

JSONB 内容必须满足：

- 有 schema 版本字段，例如 `schemaVersion`。
- 关键业务对象同时保存 `id/code/nameCn/nameEn/versionNo`。
- 生成 hash 时使用稳定序列化，避免字段顺序变化导致误判。
- 发布包、快照、事件 payload 不允许保存密钥、连接串、Token。

## 9. 完整性和性能策略

### 9.1 完整性

| 对象 | 完整性策略 |
| --- | --- |
| 编辑源数据 | 应用层校验 + 唯一索引 + 引用检查 |
| 发布包 | 发布检查通过后生成，不可变，带 `content_hash` |
| 快照 | 后端 `buildSnapshot` 生成，不信任前端本地计算 |
| 审核记录 | 保存前后状态、前后 hash、检查结果 JSON |
| 同步事件 | 发布包写入成功后写 outbox，同事务提交 |

### 9.2 性能

| 查询场景 | 优先读取 |
| --- | --- |
| 产品中心首页工作台 | `pc_read_model` + `pc_gap_task` |
| 销售只读产品详情 | `pc_published_package` |
| 报价/订单配置器初始化 | `pc_published_package` |
| 订单历史复原 | `order_product_snapshot` |
| ERP/MES 同步 | `pc_event_outbox` + `pc_published_package` |
| AI 总览和解释 | `pc_read_model.ai_context_json` |

避免模式：

- 订单保存时 join 草稿问题、答案、组件、价格多张表。
- 历史订单展示实时读取当前产品名称和当前价格。
- AI 每次临时 join 配置、价格、资料、BOM 后再拼上下文。
- 发布检查只保存“通过/失败”，不保存检查明细。

## 10. 后续 SQL 注释规范

后续写 PostgreSQL migration 时，必须使用中文注释覆盖所有表和所有字段。

示例仅表示规范，不是本次要执行的 SQL：

```sql
COMMENT ON TABLE pc_product_model IS '产品模型表';
COMMENT ON COLUMN pc_product_model.product_model_id IS '产品模型ID';
COMMENT ON COLUMN pc_product_model.product_model_code IS '产品模型编码';
COMMENT ON COLUMN pc_product_model.product_model_name_cn IS '产品模型中文名称';
COMMENT ON COLUMN pc_product_model.product_model_name_en IS '产品模型英文名称';
COMMENT ON COLUMN pc_product_model.create_time IS '创建时间，UTC timestamptz';
COMMENT ON COLUMN pc_product_model.update_time IS '更新时间，UTC timestamptz';
```

落 SQL 前检查清单：

- 每张表都有 `COMMENT ON TABLE`。
- 每个字段都有 `COMMENT ON COLUMN`。
- 每个时间字段注释包含 `UTC timestamptz`。
- 平台功能表是否加入租户忽略表已经逐表确认。
- 如果保留 `tenant_id`，必须说明平台数据固定 `tenant_id = 1`，且不能使用 `tenant_id = 0`。
- 所有 `jsonb` 字段注释说明保存内容。
- 所有状态字段注释列出枚举含义。
- 所有 hash 字段说明 hash 对象来源。

## 11. 第一阶段落地顺序建议

第一阶段不要一次性把所有表都实现成完整后台。建议按能跑通闭环的顺序：

1. 基础信息：`pc_category`、`pc_material`、`pc_component`、`pc_media_asset`、`pc_media_binding`
2. 配置编辑：`pc_product_model`、`pc_sales_variant`、`pc_config_template`、`pc_config_template_version`、问题/答案/规则表
3. 价格引用：`pc_price_plan`、`pc_price_plan_version`、`pc_price_rule_item`
4. 发布闭环：`pc_publish_check`、`pc_approval_record`、`pc_published_package`
5. 快照和读取：`order_product_snapshot`、`pc_read_model`
6. 同步预留：`pc_event_outbox`

其中 `pc_published_package`、`order_product_snapshot`、`pc_read_model` 是性能和可追溯的关键，不应省略。

## 12. 暂不确定项

| 待确认项 | 当前建议 |
| --- | --- |
| 产品中心表是否全部加入租户忽略表 | 倾向加入，因为用户已明确这是平台功能 |
| 价格是否存在商户专属价 | 第一版按平台价；后续通过 `price_scope` 扩展 |
| 老库价格矩阵是否迁移 | 第一版不重导，先用 `pc_price_rule_item.calculation_mode = ADAPTER` 包装 |
| 订单快照表具体放在哪个模块 | 属于订单模块，但字段结构按本文执行 |
| 规则 JSON schema | 技术设计阶段单独补规则 schema 文档 |
| read model 刷新方式 | 第一版发布时同步刷新，后续可改异步事件刷新 |

## 13. 一句话总结

这套数据库不是传统三范式优先，也不是所有东西乱塞宽表。正确边界是：

```text
源数据清晰
发布包冻结
订单快照自包含
读模型拉宽
同步事件可追踪
所有表字段中文注释完整
```
