# 产品设计中心模块定义与菜单重构草案

创建日期：2026-06-08

状态：动态草案，后续按模块逐步补充、校正、确认。

适用范围：基础资料、产品设计、配置规则、组件/BOM、价格报价、发布应用、导入同步等产品能力相关模块。

## 1. 文档目标

这个文档用于重新梳理“产品设计中心/共享产品能力中心”的业务边界和菜单结构，避免系统实现变成一堆彼此割裂的 CRUD 页面。

当前要先明确三件事：

1. 每个模块到底是干什么的。
2. 每个模块支撑哪个业务流程、服务哪个下游系统。
3. 菜单应该如何重新分层，让用户按业务逻辑使用系统，而不是按数据表找页面。

后续修改代码前，优先在本文档里确认模块定义和流程，再进入具体实现。

## 2. 核心原则

### 2.1 基础资料和产品设计要分开

基础资料是可复用的主数据，例如产品分类、物料、单位、资料资产等。

产品设计是围绕某个产品模型展开的配置、组件、BOM、价格、发布流程。

不能把所有页面都堆在“产品设计中心”里，也不能把真正的产品设计流程拆散成纯表维护。

### 2.2 凡是可采购、可库存、进 ERP 的东西，都应该是物料

物料是采购、库存、成本、ERP、MES 的基础对象。

如果某个辅材需要单独采购、单独入库、单独同步 ERP，它就应该作为物料维护，而不是只存在于组件库里。

“辅材”更适合作为物料类型，例如：

- 主材
- 辅材
- 包材
- 五金件
- 成品
- 半成品
- 服务项

### 2.3 组件库不是采购主数据，而是设计和 BOM 抽象层

组件库的职责不是替代 ERP 物料，而是把配置选择翻译成可复用的组件/BOM 结构。

例如“电机套件”可以是一个组件，但它背后可能包含：

- 电机物料
- 遥控器物料
- 连接件物料
- 包装物料

所以真实业务里，一个组件通常应该支持一对多物料明细，而不是只绑定一个物料。

### 2.4 发布后的产品能力必须可追溯

发布包应该是下游系统使用的稳定版本。

订单、报价、ERP、MES 不应该直接依赖正在编辑中的配置数据，而应该使用已发布版本或快照。

## 3. 推荐业务主流程

```text
基础资料准备
  -> 产品模型定义
  -> 销售变体定义
  -> 配置模板/问题/选项设计
  -> 组件/BOM/资料绑定
  -> 价格方案与报价预览
  -> 缺口检查
  -> 审批发布
  -> 销售只读、报价、订单、ERP/MES 同步
```

这个流程里，基础资料是前置能力，产品设计是核心工作流，发布与同步是对外生效边界。

## 4. 模块定义

### 4.1 基础资料库

基础资料库用于维护可复用的产品主数据。它的目标是让后续产品设计、配置、价格、发布都能引用统一数据。

| 模块 | 作用 | 支撑什么 | 目标 |
| --- | --- | --- | --- |
| 产品分类 | 维护产品业务分类、层级、路径、状态 | 产品模型归类、销售展示、报表分析、发布范围 | 让产品按业务分类组织，而不是散落维护 |
| 物料管理 | 维护 ERP/采购/库存意义上的物料主数据 | BOM、成本、采购、库存、ERP、MES | 所有可采购、可库存、可核算对象统一归口 |
| 单位维护 | 维护产品域常用单位，例如 cm、inch、m2、PC、Set | 物料数量、组件用量、价格规则、报价展示 | 避免单位散落在字典里，业务用户找不到 |
| 资料资产 | 维护图片、手册、文件、文档等资料 | 销售展示、配置说明、发布包完整性检查 | 让产品资料可复用、可绑定、可追溯 |
| 资料绑定 | 把资料绑定到分类、模型、变体、组件、物料或销售场景 | 销售只读页、报价页、发布包 | 明确资料属于谁、用于哪里 |

#### 当前需要注意

当前系统里单位主要通过字典 `product_unit` 维护。业务上这类内容应该有清晰入口，可以仍然复用字典能力，但菜单和页面概念上要让用户知道“单位在哪里维护”。

### 4.2 物料管理

物料管理是采购、库存、成本和 ERP/MES 的主数据入口。

物料关注的是“真实的东西”：

- 是否可以买
- 是否入库
- 是否有供应商
- 是否需要成本核算
- 是否同步 ERP/MES
- 使用什么单位

物料不负责表达“用户选择了哪个配置后要带出什么组合”，这个职责属于组件/BOM 或配置规则。

#### 典型字段

| 字段 | 说明 |
| --- | --- |
| material_code | 物料编码 |
| material_name_cn / material_name_en | 物料名称 |
| material_type | 物料类型，例如主材、辅材、包材、成品、半成品 |
| business_type | 业务类型 |
| unit_code | 单位 |
| supplier_name | 供应商 |
| status | 状态 |

#### 关键结论

辅材如果需要采购，就应该首先是物料。不要把“辅材管理”做成独立于物料之外的采购主数据，否则 ERP 来源、库存来源、成本来源都会乱。

### 4.3 组件库

组件库是产品设计和配置 BOM 的中间层。

组件关注的是“一个配置选择或一个产品结构单元，应该带出哪些物料和用量”。

它支撑：

- 配置选项自动带出物料
- BOM 展开
- 报价成本估算
- ERP/MES 同步前的数据转换
- 产品发布检查

#### 组件库不应该做什么

组件库不应该替代物料主数据。

组件库不应该单独成为采购来源。

组件库不应该让同一个真实物料在多个地方重复维护。

#### 组件库应该做什么

| 场景 | 组件库职责 |
| --- | --- |
| 一个选项对应一个物料 | 组件可以直接映射一个物料 |
| 一个选项对应多个物料 | 组件应维护组件明细，一对多映射物料 |
| 一个结构单元可复用 | 组件作为可复用 BOM 片段 |
| 用量随尺寸变化 | 组件规则或 BOM 规则计算数量 |
| 市场或变体不同 | 组件绑定适用范围 |

#### 当前实现风险

当前组件表更像“组件绑定单个物料的快照”。这对简单场景可以工作，但对真实组件库不够。

后续如果要支撑组件包含多个物料，可能需要补充组件明细结构，例如 `component_material` 一类的关系。这个属于数据库结构调整，后续必须单独评审和规划，不能直接改。

### 4.4 产品模型

产品模型用于定义“这是一个什么产品”。

例如 Roller Shade、Venetian Blind、Curtain Track 等，都应该先成为产品模型。

产品模型支撑：

- 销售变体
- 配置模板
- 价格方案
- 资料绑定
- 发布包
- 销售只读页

目标是让所有后续配置都围绕一个明确产品展开。

### 4.5 销售变体

销售变体用于表达同一个产品模型在不同市场、渠道、等级、控制方式、包装方式下的销售版本。

它支撑：

- 市场差异
- 价格差异
- 可选配置差异
- 发布范围差异
- 销售端展示差异

销售变体不是物料，也不是成品 SKU 的简单替代。它更像销售能力维度。

### 4.6 配置模板、问题组、问题、选项

这些模块共同定义“销售或下单时用户需要怎么选”。

| 模块 | 作用 | 支撑什么 |
| --- | --- | --- |
| 配置模板 | 定义某个模型/变体使用哪套配置结构 | 报价、下单、销售展示 |
| 问题组 | 把问题按业务步骤分组 | 配置页面分段、复用 |
| 问题 | 用户需要回答的配置项 | 输入、选择、校验 |
| 选项 | 问题可选值 | 组件带出、价格、规则 |

这个区域的核心目标是让产品配置可维护、可复用、可发布，而不是写死在前端页面里。

### 4.7 配置规则

配置规则用于处理问题之间、选项之间、选项和组件之间的关系。

它支撑：

- 显示/隐藏
- 必填/非必填
- 互斥/依赖
- 默认值
- 兼容性检查
- 自动带出组件或物料
- 用量计算

如果没有配置规则，系统只能做到“填表式配置”，不能支撑真实产品设计。

### 4.8 价格与报价

价格与报价用于把产品配置结果转成销售价格、成本估算和报价预览。

| 模块 | 作用 | 支撑什么 |
| --- | --- | --- |
| 价格方案 | 定义某个模型/变体使用的价格体系 | 报价、销售、渠道价 |
| 价格规则 | 维护基础价、选项价、尺寸价、组件价、公式价 | 报价计算 |
| 价格试算 | 用测试配置验证价格结果 | 发布前验证 |
| 报价预览 | 模拟销售报价页面 | 业务验收、销售确认 |

价格模块不应该独立于配置结果存在，否则无法解释价格从哪里来。

### 4.9 发布与应用

发布与应用是产品能力对外生效的边界。

| 模块 | 作用 | 支撑什么 |
| --- | --- | --- |
| 缺口待办 | 展示发布检查中发现的阻断项和预警项 | 发布前补齐数据 |
| 测试发布 | 在不正式生效的情况下验证模型、配置、价格、资料 | 业务验收 |
| 审核审批 | 控制产品能力从草稿到发布 | 权限、流程、合规 |
| 发布包 | 生成不可变版本 | 销售、报价、订单、ERP/MES |
| 同步日志 | 跟踪同步到下游系统的结果 | 运维、追溯 |
| 销售只读总览 | 以销售端视角查看已发布能力 | 销售、经销商门户、客户门户 |

发布后的数据应该稳定、可追溯、可被下游消费。

### 4.10 导入中心

导入中心用于批量导入产品能力相关数据。

它支撑：

- 物料导入
- 分类导入
- 模型导入
- 配置导入
- 价格导入
- 资料绑定导入

导入中心不是一个孤立功能，它应该服务于基础资料和产品设计流程。

导入结果必须能看到：

- 批次
- 文件
- 解析状态
- 行级问题
- 目标对象
- 是否已落库
- 是否影响发布检查

## 5. 菜单重构建议

下面是初版菜单分层建议，用于后续讨论，不代表马上修改代码。

### 5.1 基础资料

用于维护产品设计前必须存在的主数据。

```text
基础资料
  产品分类
  物料管理
  单位维护
  资料资产
  资料绑定
```

说明：

- “辅材”如果是可采购对象，应通过物料类型维护在物料管理里。
- 单位可以底层复用字典，但业务菜单要让用户能找到。
- 资料绑定也可以后续按场景拆到设计流程里，但基础入口需要存在。

### 5.2 产品设计

用于围绕产品模型完成配置和组件设计。

```text
产品设计
  产品模型
  销售变体
  配置模板
  问题组
  问题与选项
  配置规则
  组件库
  组件/BOM 绑定
```

说明：

- 组件库建议放在产品设计里，因为它不是采购主数据，而是设计/BOM 抽象层。
- 如果业务用户强烈认为组件库属于基础资料，也可以在基础资料中保留入口，但概念上要说明它不是物料来源。

### 5.3 价格与报价

用于维护价格方案、验证报价结果。

```text
价格与报价
  价格方案
  价格规则
  价格试算
  报价预览
```

说明：

- 报价预览应该从产品模型/销售变体/配置模板进入，而不是孤立页面。
- 价格规则必须能追溯到配置选项、组件、尺寸或公式。

### 5.4 发布与应用

用于发布前检查、审批、生成发布包和同步下游。

```text
发布与应用
  设计工作台
  缺口待办
  测试发布
  审核审批
  发布包
  同步日志
  销售只读总览
```

说明：

- 设计工作台应该是流程入口，而不是单纯统计首页。
- 缺口待办应能跳转到对应维护页面。
- 发布包是下游系统使用的稳定版本。

### 5.5 导入与工具

用于批量导入和辅助检查。

```text
导入与工具
  导入中心
  导入批次
  行级问题
  变更影响分析
```

说明：

- 导入中心可以独立成组，也可以挂到发布与应用下。
- 如果导入是高频操作，建议独立成组。

## 6. 当前实现偏差记录

这部分用于记录目前观察到的业务偏差，后续逐项确认后再改。

| 问题 | 当前表现 | 建议方向 |
| --- | --- | --- |
| 产品设计中心缺少流程主线 | 页面像分散 CRUD，没有围绕模型推进 | 增加模型驱动的设计工作台和流程入口 |
| 辅材/组件概念混淆 | “辅材管理”容易被理解成采购主数据 | 可采购辅材归物料，组件库归设计/BOM |
| 组件库能力不足 | 当前更像单组件绑定单物料 | 后续评审是否需要组件明细，一对多物料 |
| 单位入口不明显 | 单位在系统字典里，业务用户不容易找到 | 增加产品域单位维护入口或说明 |
| 报价预览入口孤立 | 预览和模型/配置/价格关系不够明显 | 从模型或价格方案进入预览 |
| 发布模块像功能列表 | 缺少从检查到审批到发布的流程感 | 以发布流程组织菜单和页面 |

后续每确认一个模块，建议按以下结构补充：

```text
模块名称：

权限边界：

```

## 8. 待确认问题

1. 组件库最终放在“基础资料”还是“产品设计”，是否需要两个入口但同一页面。
2. 当前 `pc_component` 是否要升级为组件主表 + 组件物料明细表。
3. 单位维护是否只复用系统字典，还是增加产品域页面。
4. 价格规则是否需要区分销售价、成本价、渠道价、市场价。
5. 发布包是否需要包含模型、变体、配置、组件、物料快照、资料、价格的完整快照。
6. ERP/MES 同步边界是同步发布包，还是同步展开后的 BOM/物料清单。
7. 导入中心是独立一级菜单，还是放在“导入与工具”或“发布与应用”下。

## 9. 工作约定

后续先在本文档里把模块定义、业务流程、菜单结构顺清楚，再进入代码修改。

每次修改实现前，需要先确认：

- 改的是哪个模块。
- 这个模块在本文档里的定义是否已经确认。
- 是否涉及数据库结构、权限、菜单、i18n、导入、发布包、ERP/MES 同步。
- 如果涉及数据库结构或跨模块调整，必须单独规划，不直接动。
