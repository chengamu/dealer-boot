# 共享产品能力中心数据库落库评审

## 结论

当前 `sql/postgresql/product_capability.sql` 已完成第一轮落库前硬规则检查，并已在一次性 PostgreSQL 16 空库容器中完成 `base.sql + product_capability.sql` 顺序试执行。

本轮已修正：

- `pc_config_template_version` 索引从不存在的 `status` 字段改为 `version_status`。
- `pc_price_plan_version` 索引从不存在的 `status` 字段改为 `version_status`。
- Batch 2 / Batch 3 追加的产品能力菜单 SQL 补齐 `tenant_id = 1`，避免 `sys_menu.tenant_id` 非空约束失败。
- 最后一段 `sys_role_menu` 授权 SQL 补齐 `tenant_id = 1`，匹配现有系统表结构。
- 23 张 `pc_*` 平台共享表已加入 `bocoo.tenant.ignore-tables`，避免租户拦截器追加不存在的 `tenant_id`。

历史临时库执行结果：

- `pc_*` 表：23 张。
- 产品能力菜单：52 条。`24002/24003/24004` 为旧聚合入口，保留但隐藏；`24100-24106` 为基础资料独立二级入口。
- `role_id = 1` 产品能力授权：52 条。

第一阶段菜单归一化后需要重新执行真实库核验：

- 正式菜单范围改为 `24200-24499`。
- 预期正式产品能力菜单和按钮共 `86` 条，旧 `24000-24199` 只作为历史草案范围。
- 正式一级菜单为 `基础资料`、`配置与价格`、`发布与应用`，二级入口以入口文档第 4 节为准。
- `product_material_type`、`product_component_type`、`product_business_type`、`product_unit` 四类字典必须落库。

## 租户策略

产品能力中心当前按平台共享能力设计，核心业务表不加 `tenant_id`。

运行时要求：

- 所有 `pc_*` 表必须配置到 `bocoo.tenant.ignore-tables`。
- 产品能力核心查询不依赖当前订单租户。
- 订单、ERP、MES 消费产品能力时，调用通用快照实例 API 并保存 `snapshot_id/snapshot_hash` 或少量展示冗余，不反向修改产品能力核心表。
- `sys_menu`、`sys_role_menu` 仍按现有系统表结构保留 `tenant_id = 1`。

## UTC 规则

所有产品能力时间字段使用 PostgreSQL `timestamptz`，字段注释标明 UTC 语义。

落库和开发要求：

- 后端写入时间继续使用 `TimeUtils.utcNow()`。
- 前端展示后端时间继续使用项目 UTC 工具。
- SQL 中菜单初始化使用 `now()` 仅用于系统菜单种子数据；业务表不把 `now()` 作为权威默认值。

## 注释规则

当前 23 张 `pc_*` 表均已检查：

- 每张表都有 `COMMENT ON TABLE` 中文注释。
- 每个字段都有 `COMMENT ON COLUMN` 中文注释。
- 时间字段注释包含 UTC 或 `timestamptz` 说明。

后续新增字段必须同步补中文注释，不能只补关键字段。

## 索引规则

当前已做自动检查：

- 所有 `CREATE INDEX` / `CREATE UNIQUE INDEX` 引用的表存在。
- 所有索引引用的字段存在。
- 软删除唯一索引使用 `WHERE COALESCE(del_flag, '0') = '0'` 保留历史数据重建空间。

后续 DBA 评审建议重点关注：

- `jsonb` 字段是否需要 GIN 索引。
- 发布包、outbox、价格规则、配置规则是否需要按实际查询增加组合索引。
- 销售只读视图后续如果从发布包 read model 展开，是否需要独立 read model 表或物化视图。

## 仍需确认

- 当前 SQL 仍是初始化草案，不是生产 migration。
- 真实开发库或生产库执行前仍需确认目标库是否已有同名表、菜单 ID、权限码。
- 发布包 JSON/hash schema 仍是 MVP，需要后续冻结完整 read model 结构。
- 快照持久化已统一放在产品能力中心 `pc_product_snapshot_instance`；来源系统侧仍需在订单、ERP、MES 业务表中保存 `snapshot_id/snapshot_hash` 引用。
