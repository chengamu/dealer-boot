> 本文是功能全集和模块拆分清单。开发入口、实施顺序、验收门和 `done` 状态口径统一看 [共享产品能力中心开发入口.md](./共享产品能力中心开发入口.md)。

> 这份文档不是数据库建表，也不是页面原型说明。  
> 它的作用是把现有设计拆成开发可以排期的功能模块：在当前 `base-boot` 项目已有后台能力基础上，要实现共享产品能力中心，第一版还需要补哪些页面、哪些小功能、哪些业务能力。

## 0. 当前项目适配原则

当前项目已经具备一批通用基础能力，排期时不要把它们当成从零开发：

| 能力 | 当前处理 |
| --- | --- |
| 登录、角色、菜单、按钮权限 | 复用现有 Sa-Token、系统菜单、角色授权和前端权限控制 |
| 租户、审计、数据权限 | 复用现有租户上下文、审计字段、MyBatis-Plus 和数据权限扩展 |
| i18n | 新增可见文案只写 `i18n/locales/en_US.json`，同步到前后端运行时 JSON |
| 文件上传/图片资料 | 复用现有 OSS/文件管理，配置中心只新增资料资产业务绑定和版本 |
| 字典/标准数据 | 状态、类型、单位、规则类型优先走现有字典或标准数据能力 |
| 前端布局和样式 | 复用 `admin-ui` 现有 Element Plus 管理端风格，不照搬 H5 原型 |
| 后端 CRUD 形态 | 复用现有代码生成器、Controller/Service/Mapper/VO/BO/Page/R 习惯 |

普通 grid/list、特殊自定义页面例外、前后端权限、导出、i18n、UTC 和代码生成器落地规范统一参考 [fullstack-code-standards.md](../项目配置和代码风格/fullstack-code-standards.md)。

因此，下面所有 P0 不表示重做通用后台，而表示必须补齐对应业务能力。

这份清单里的基础信息、配置中心、辅材管理、资料资产、价格中心、规则、用量、BOM、发布检查都属于共享产品能力，不是订单系统私有能力。当前可以寄居在订单系统代码库中，但要按未来独立服务设计边界。

数据库逻辑边界以 [共享产品能力中心数据库设计草案.md](./共享产品能力中心数据库设计草案.md) 为准。接口和后台 Java 落地边界以 [共享产品能力中心API与后端实现约束.md](./共享产品能力中心API与后端实现约束.md) 为准。后续页面设计、接口设计和技术实施计划都应基于这两份文档继续细化，但本文仍只做功能拆分，不直接生成 migration。

进入编码前还需要补一份技术实施计划，至少确认：

| 待确认项 | 为什么重要 |
| --- | --- |
| 后端模块落点 | 建议独立业务模块，不把共享产品能力塞进 `bocoo-modules-system` |
| 数据库边界 | 参考数据库设计草案；当前可同库，未来应支持独立库；表前缀、外键和接口要按可拆分设计 |
| 数据同步策略 | 后期产品能力中心发布版本后，同步只读发布包或 read model 到订单、ERP、MES |
| 平台功能和租户拦截 | 产品能力核心数据按平台共享主数据设计；技术落地需确认加入租户忽略表或统一平台租户 |
| UI i18n 与业务多语言 | UI 文案走 i18n JSON；产品名、答案名、客户说明是业务字段和快照内容 |
| 后端求值权威 | 前端只预览，发布、报价、订单保存必须以后端 evaluate/buildSnapshot 为准 |
| PostgreSQL/UTC | 新表和时间字段必须符合当前 PG + UTC 口径，不补 MySQL 新 SQL |
| 老库适配器 | 旧价格和旧配置只读包装，不把旧表结构直接变成新核心模型 |

## 1. 总体模块

第一版建议拆成 7 个业务模块：

| 模块 | 作用 | 第一版优先级 |
| --- | --- | --- |
| 基础信息中心 | 维护产品域分类树、物料、组件、资料绑定、单位/字典扩展等公共基础 | P0 |
| 配置中心 | 建产品模型、销售变体、问题组、问题、答案、规则、用量引用 | P0 |
| 价格中心 | 维护销售价格方案、矩阵价、工艺价、固定组件价、运费、价格测试 | P0 |
| 产品审核发布 | 联合检查配置、价格、资料、组件、测试，发布可销售版本 | P0 |
| 销售只读展示 | 给销售/领导看完整产品配置、问题、价格状态、资料和限制 | P1 |
| AI 配置助手 | 生成草稿、推荐问题组、发现缺口、解释字段、打开预览 | P1 |
| 系统支撑能力 | 权限、版本、草稿、影响分析、缺口待办、日志、快照 | P0 |

这些模块逻辑上属于同一个“共享产品能力中心”。订单系统只消费发布版本和快照，不直接拥有这些配置数据。

上表是业务能力域，不是侧边栏菜单。正式菜单按当前后台“两级菜单最好用”的习惯组织为三个一级菜单：`基础资料`、`配置与价格`、`发布与应用`。每个一级菜单下面直接挂二级业务入口，不再放一个笼统的“产品能力 / 产品中心”承载全部功能，也不新增三级菜单。

菜单归类建议：

| 一级菜单 | 二级入口 |
| --- | --- |
| 基础资料 | 产品分类、物料管理、辅材管理、资料资产、资料绑定、产品模型、销售变体 |
| 配置与价格 | 配置模板、问题组模板、价格中心、报价预览、导入中心 |
| 发布与应用 | 工作台、测试发布、审核审批、缺口待办、发布包、同步日志、销售只读总览 |

暂缓但要留边界的模块：

| 模块 | 当前处理 |
| --- | --- |
| 订单侧详细设计 | 暂不展开，只保留配置快照、价格快照、订单读取接口边界 |
| ERP/生产侧详细设计 | 暂不展开，只保证物料、组件、用量、BOM 输出口径清楚 |
| 导入中心 | 第一版保留菜单、批次、解析预览和问题清单；完整资料包自动导入、AI 映射和批量提交后置 |
| 独立服务拆分 | 后置，但当前开发要保留 API 边界、同步包和 read model 设计空间 |

## 2. 第一版主流程

开发实现时，主流程按下面闭环设计：

```text
基础信息准备
  -> 建产品分类 / 产品模型 / 销售变体
  -> 挂问题组
  -> 维护配置问题和答案
  -> 答案绑定组件 / 资料 / 规则
  -> 配置中心引用价格方案
  -> 价格中心发布价格方案
  -> 配置测试 / 价格测试 / BOM 预览 / 资料检查
  -> 缺口待办补齐
  -> 产品审核发布
  -> 销售只读展示
  -> 报价/订单读取发布版本并保存快照
```

第一版可销售条件：

```text
配置模板已发布
+ 价格方案已发布
+ 客户可见资料齐全
+ 组件绑定齐全
+ 测试用例通过
+ 无 BLOCKER
= 商品可销售
```

## 3. 基础信息中心

基础信息中心不是一个小栏目，它是配置中心能运转的地基。凡是会被多个产品、价格、订单、生产共用的东西，都不要写死在单个产品表单里。

这里的基础信息中心是“产品配置域基础信息”，不是替换系统管理已有的部门、角色、字典、OSS、标准数据。能复用的用现有能力；只有产品域特有的分类、物料、组件、资料绑定、用量单位口径需要新增业务表和页面。

### 3.1 产品分类树

用途：

- 管一级类目、二级产品形态。
- 支持导航、筛选、报表、权限。
- 不承载控制方式、罩壳、面料特性等后续配置维度。

功能清单：

| 小功能 | 说明 | 优先级 |
| --- | --- | --- |
| 分类树维护 | 新增、编辑、停用一级/二级分类 | P0 |
| 分类编码 | 维护 code、中文名、英文名、排序、状态 | P0 |
| 分类引用检查 | 停用分类前提示挂载了哪些产品模型 | P0 |
| 分类权限预留 | 后期可按分类控制产品运营权限 | P1 |

界面建议：

- 左侧树：一级/二级分类。
- 右侧表单：编码、中文名、英文名、排序、状态。
- 底部引用区：当前分类下产品数量、是否可停用。

### 3.2 物料管理

用途：

- 统一管理后期会被配置、价格、BOM、ERP 读取的物料口径。
- 不做采购审批、不扣库存，但要提前分清物料类型。

物料类型至少包含，P0 通过 `product_material_type` 字典维护：

| 字典值 | 物料类型 | 例子 | 影响模块 |
| --- | --- | --- | --- |
| `FABRIC` | 面料/布匹 | YZB201、YB180002、XL241501 | 配置选项、用量、价格、BOM、库存 |
| `COLOR` | 颜色/色卡 | White、Beige、Night Black | 配置答案、图片、销售展示 |
| `MOTOR` | 电机 | Lithium Battery Motor、Bluetooth Motor | 组件绑定、固定组件价、BOM |
| `REMOTE` | 遥控器 | 单通道、多通道遥控器 | 标准品、选配价、订单提示 |
| `HUB` | Hub/网关 | Mini Bridge、Hub | 标准品、组件价、资料说明 |
| `SOLAR_PANEL` | 太阳能板 | Solar Panel | 标准品、自动带出、固定价 |
| `PROFILE` | 型材/铝杆 | 罩壳、侧轨、轨道 | 用量模型、BOM、成本 |
| `BOTTOM_RAIL` | 下杆/底杆 | 加重下杆、圆形底杆、方形底杆 | 用量模型、BOM、成本、销售展示 |
| `ACCESSORY` | 辅材/配件 | 链条、支架、螺丝、安装包 | BOM、固定数量规则 |
| `PACKAGING` | 包装件 | 纸箱、护角、泡棉 | BOM、包装规则 |
| `SERVICE` | 服务项 | 测量、安装、加急 | 价格中心、订单，不进库存 |
| `FINISHED_GOOD` | 标准成品 | 遥控器、充电器、样品、配件包 | 可单独销售、固定价、库存 |

功能清单：

| 小功能 | 说明 | 优先级 |
| --- | --- | --- |
| 物料列表 | 按物料类型、状态、供应商、编码搜索 | P0 |
| 物料基础信息表单 | 编码、中文名、英文名、类型、单位、状态、供应商名 | P0 |
| 后期物料口径 | 标记成品、定制成品、原材料、采购组件、辅材、服务项 | P0 |
| 面料属性 | 幅宽、克重、遮光度、系列、颜色、方向、库存单位 | P0 |
| 电机属性 | 型号、扭矩等级、适用管径、适用宽高/重量、供电方式 | P0 |
| 型材属性 | 型材类型、长度单位、颜色、扣减规则引用 | P1 |
| 物料状态 | 草稿、启用、停用、需复核 | P0 |
| 引用检查 | 查看被哪些产品、答案、组件、价格方案引用 | P0 |

界面建议：

- 左侧筛选：物料类型、状态、供应商、是否被引用。
- 中间表格：编码、名称、类型、后期口径、状态、引用次数。
- 右侧抽屉：基础信息、属性、图片资料、价格引用、影响分析。

### 3.3 辅材管理

用途：

- 菜单显示为“辅材管理”，业务对象包含配置答案可以自动带出的组件、辅材、标准配件和可选配件。
- 组件必须来自物料基础信息，不能在答案里手工写一段文本。

功能清单：

| 小功能 | 说明 | 优先级 |
| --- | --- | --- |
| 辅材列表 | 电机、遥控、Hub、太阳能板、面料、型材、下杆、支架、链条、包装件、样品等 | P0 |
| 组件绑定物料 | 每个组件关联一个物料或标准品 | P0 |
| 组件适用范围 | 适用产品分类、产品模型、销售变体、问题/答案 | P0 |
| 组件默认数量 | 固定数量、按规则数量、按用量模型数量 | P0 |
| 组件资料 | 图片、说明、安装/使用说明 | P0 |
| 组件价格引用 | 固定组件价、包含在基础价、单独加价 | P0 |
| 组件影响分析 | 修改组件前显示影响哪些产品、答案、BOM | P0 |

辅材类型至少包含，P0 通过 `product_component_type` 字典维护：

| 字典值 | 辅材类型 | 例子 |
| --- | --- | --- |
| `MOTOR` | 电机 | Lithium Battery Motor |
| `REMOTE` | 遥控器 | 单通道、多通道遥控器 |
| `HUB` | Hub/网关 | Mini Bridge |
| `SOLAR_PANEL` | 太阳能板 | Solar Panel |
| `BOTTOM_RAIL` | 下杆/底杆 | 加重下杆、圆形底杆 |
| `BRACKET` | 支架 | 安装支架、侧装支架 |
| `CHAIN` | 链条 | 珠链、拉绳 |
| `TUBE` | 卷管 | 38 管、50 管 |
| `PROFILE` | 型材 | 罩壳、侧轨 |
| `FABRIC` | 面料组件 | 默认面料包、样布 |
| `ACCESSORY` | 配件 | 螺丝包、安装包 |
| `PACKAGING` | 包装件 | 纸箱、护角 |
| `SAMPLE` | 样品 | 色卡、样布、样品包 |

界面建议：

- 辅材管理不要塞在产品页里，只在产品页显示引用摘要。
- 组件详情页用 Tab 或分区：基础信息、适用范围、资料、价格引用、影响分析。
- 从配置答案绑定组件时，用搜索选择组件，不允许自由输入组件名。

### 3.4 资料资产库

用途：

- 管产品图、面料图、颜色图、组件图、安装说明、测量说明、客户提示、规则说明。
- 客户反馈里和提示/说明有关的内容，沉淀到这里。
- 文件本体复用现有 OSS/文件管理；资料资产库只负责绑定对象、资料用途、语言、版本、发布检查和快照引用。

功能清单：

| 小功能 | 说明 | 优先级 |
| --- | --- | --- |
| 资料上传/维护 | 图片、PDF、说明文本、安装/测量帮助 | P0 |
| 资料绑定对象 | 产品、问题、答案、组件、规则、标准品 | P0 |
| 中英文说明 | 客户可见内容必须有中英文完整性检查 | P0 |
| 图片 alt/用途 | 颜色图、组件图、说明图要有用途和 alt | P0 |
| 资料版本 | 图片和说明文字更新后保留版本 | P0 |
| 发布检查 | 客户可见资料缺失时 WARNING 或 BLOCKER | P0 |
| 客户反馈转资料 | 反馈归类为提示缺口、帮助文档缺口、配件说明缺口 | P1 |

界面建议：

- 资料资产库主界面按绑定对象和资料类型筛选。
- 产品配置页只显示“资料齐全/缺资料”的摘要和入口。
- 销售只读页读取已发布资料，不允许销售改资料。

### 3.5 字典、单位和通用参数

功能清单：

说明：状态、类型、单位等优先接入当前项目已有字典/标准数据管理，不在配置中心里硬编码一套枚举页面。只有配置求值器必须稳定依赖的 code，需要在业务设计中固定命名和兼容策略。

字典读取口径：

- 页面下拉通过 `/system/dict/data/type/{dictType}` 读取；该接口对非公开字典检查登录态，不按菜单按钮权限过滤，登录后业务页面可用。
- 字典维护仍需要系统管理字典权限；普通产品配置用户只读使用。
- 面料、电机、遥控、下杆、辅材等类型只作为分类、筛选、导入校验和展示翻译；物料名称、编码、属性、供应商、适用范围、价格和 BOM 引用仍在业务表维护。

| 小功能 | 说明 | 优先级 |
| --- | --- | --- |
| 单位维护 | inch、cm、m、m2、PC、Set | P0 |
| 内部计算单位 | 配置求值器内部统一 cm | P0 |
| 显示单位 | 销售/客户页面可显示 inch | P0 |
| 状态字典 | 草稿、待审、已发布、停用、需复核 | P0 |
| 物料类型字典 | `product_material_type`：面料、颜色、电机、遥控、Hub、太阳能板、型材、下杆、辅材、包装件、服务项、标准成品 | P0 |
| 辅材类型字典 | `product_component_type`：电机、遥控、Hub、太阳能板、下杆、支架、链条、卷管、型材、面料组件、配件、包装件、样品 | P0 |
| 后期物料口径字典 | `product_business_type`：成品、定制品、原料、采购件、服务 | P0 |
| 变体维度字典 | control_method、market、grade、package_type 等 | P0 |
| 规则类型字典 | 必填、显示隐藏、禁选、尺寸、用量、自动组件等 | P0 |

## 4. 配置中心

配置中心只管产品能力定义，不管完整销售价格、不处理订单组合、不做库存采购财务。

### 4.1 产品模型

功能清单：

| 小功能 | 说明 | 优先级 |
| --- | --- | --- |
| 产品模型列表 | 按分类、状态、销售模式、配置复杂度搜索 | P0 |
| 产品模型基础 | 分类、系列、产品族/品类、结构/外壳、销售模式、单位 | P0 |
| 产品性质 | 定制成品、标准品、半标准品、服务项 | P0 |

## 11. 第一版功能边界

### 11.1 第一版必须做

- 基础信息：产品域分类树、物料管理、辅材管理、资料资产绑定、单位/字典扩展。
- 配置中心：产品模型、销售变体、问题组、问题、答案、规则、用量、BOM 预览。
- 价格中心：价格方案、矩阵价/工艺价/固定组件价、运费、价格测试、价格快照。
- 产品审核：联合发布检查、缺口待办、影响分析、审批发布。
- 快速操作：相似产品克隆、蓝图、批量粘贴、批量适用范围、缺口清单。
- 销售只读展示：从已发布配置自动生成。

不重复开发但必须接入：

- 现有登录、权限、菜单、按钮权限。
- 现有 i18n、请求层、前端布局和组件风格。
- 现有 OSS/文件管理。
- 现有审计字段、操作日志和 UTC 时间口径。

### 11.2 第一版只预留，不展开

- 订单侧详细设计。
- ERP/生产领料、采购、库存、成本详细流程。
- 完整资料包自动导入。
- Excel 自动识别问题、答案、组件、价格。
- Agent 自动发布或自动改价。
- 折扣、客户等级、促销价格。

## 12. 建议开发顺序

| 阶段 | 目标 | 主要功能 |
| --- | --- | --- |
| 1 | 打地基 | 分类树、物料管理、辅材管理、资料资产、单位/字典 |
| 2 | 建配置 | 产品模型、销售变体、问题组、问题、答案、组件绑定 |
| 3 | 跑规则 | 显示/隐藏、必填、选项域、禁选、尺寸、电机能力、用量/BOM 预览 |
| 4 | 接价格 | 价格方案、价格版本、矩阵/工艺/组件/运费、价格测试 |
| 5 | 做发布 | 联合发布检查、缺口待办、影响分析、审批发布 |
| 6 | 做查看 | 销售只读总览、配置预览、下单模拟验证 |
| 7 | 做 AI | ConfigAgent 生成草稿、解释字段、推荐问题组、发现缺口 |

这个顺序的核心是：先让基础资料和配置中心稳定，再接价格和发布，最后做 AI 提效。不要先做 Agent 或导入，否则会把未稳定的业务口径自动放大。
