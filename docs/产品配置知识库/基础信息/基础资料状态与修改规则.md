# 基础资料状态与修改规则

本文定义基础信息中产品分类、产品字典、单位管理、物料属性、面料系列、物料资料、组件包、组件明细、资料资产等模块的通用状态和修改边界。

## 业务目标

基础资料会被工程方案、组件包、产品配置、订单快照、采购和库存长期引用。系统需要同时满足：

- 启用资料能被后续业务选择。
- 停用资料不再进入新增业务选择器。
- 历史引用仍然能看懂原资料名称、编码和规格。
- 普通用户不会误改已经启用或被引用的关键资料。
- 主管或受控账号未来可以通过独立的超级修改动作处理特殊场景。

## 状态含义

| 状态 | 含义 | 新增业务选择器 | 普通修改 |
| --- | --- | --- | --- |
| 启用 | 已确认可用，可被工程配置、组件包等新业务引用。 | 返回 | 不允许直接修改 |
| 停用 | 不再给新业务使用，历史引用继续保留。 | 默认不返回 | 允许普通修改 |

默认状态可以是启用，但启用后不允许通过普通修改随意变更资料内容。

## 按钮和接口边界

| 动作 | 按钮 | 接口 | 权限 | 说明 |
| --- | --- | --- | --- | --- |
| 修改前检查 | 修改按钮触发前置检查 | `GET .../{id}/edit-check` | 与普通修改一致 | 判断当前资料是否允许进入普通编辑。 |
| 普通修改 | 修改 | `PUT ...` | `*:edit` | 只允许修改停用状态的数据；后端必须兜底校验。 |
| 启用 / 停用 | 状态切换 | `PUT .../change-status/{id}/{status}` | 状态切换权限或编辑权限 | 只改变是否进入新增业务选择器，不代表删除历史。 |
| 超级修改 | 超级修改 | `PUT .../super-update` | `*:superEdit` | 独立按钮、独立权限、独立接口；第一阶段只在物料资料落地，其它模块不显示。 |

不要把普通修改、超级修改、复制新资料、风险审核揉进同一个按钮或同一个保存接口。

## 修改前检查

用户点击修改时，前端先调用修改前检查接口。检查通过才打开编辑抽屉；检查不通过时直接提示原因，不让用户填完一堆字段后才保存失败。

基础返回结构建议：

```json
{
  "editable": false,
  "reason": "当前资料已启用，不可直接修改，请先停用后再编辑。",
  "status": "ENABLED",
  "impactSummary": []
}
```

第一阶段检查逻辑先保持简单：

- 启用状态：不可普通修改。
- 停用状态：可以普通修改。

后续如果加入引用影响范围，继续复用 `impactSummary`：

```json
{
  "editable": false,
  "reason": "该资料已被组件包引用，请确认影响范围。",
  "status": "ENABLED",
  "impactSummary": [
    {
      "type": "组件包",
      "code": "PACK_MOTOR",
      "name": "电机系统包"
    }
  ]
}
```

## 后端兜底

前端修改前检查只是用户体验，不是安全边界。普通修改接口必须再次校验当前资料状态：

- 启用状态：拒绝普通修改。
- 停用状态：允许普通修改。

本阶段物料资料已提供超级修改接口，用于受控账号处理启用状态物料的特殊修正；其它基础资料暂不开放超级修改入口。

## 选择器规则

新增业务选择器默认只返回启用资料。

历史详情、引用查看、订单快照、组件包已有明细等场景需要能展示停用资料，但必须明确标识状态，不能因为停用而导致历史数据无法识别。

## AI 代办规则

AI 代办新增资料时默认按启用保存，除非用户明确要求先保存为停用或草稿。

AI 代办编辑资料时必须先调用修改前检查接口：

- 可编辑：继续收集字段并提交普通修改。
- 不可编辑：向用户说明原因，不直接绕过普通修改。
- 超级修改：仅物料资料支持；必须由用户明确说出，并且当前账号具备 `product:base:superEdit` 权限后才能执行。
