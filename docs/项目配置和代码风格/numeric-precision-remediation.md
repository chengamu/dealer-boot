# 数值精度整改清单

本文件只记录尚未对齐 [数值精度、展示与舍入规范](./numeric-precision-standards.md) 的代码。

执行规则：

- 每次只处理一个整改项。
- 整改单位必须是一个完整业务功能，禁止只修改页面小数位后结束。
- 每项必须扫描前端、后端、SQL、API、测试、Excel、PDF 和邮件。
- 允许统一数据库字段、计算精度、舍入边界、录入校验、编辑回显和展示格式。
- 公式计算内核可以修正，公式录入方式、变量、中文别名、函数名、条件配置和保存文本不得改变。
- 不做测试数据和旧快照兼容；结构与计算规则调整后清理并重新生成测试数据。
- 每项完成时必须同步对应知识库，只记录最终实现。
- 每项必须完成编译、自动化测试和真实页面验证。
- 人工验证必须覆盖录入、编辑回显、计算或测算、快照、导出和历史查看；不具备的环节明确标记不适用。
- 完成后删除对应整改项，不保留完成记录。
- 所有整改项删除后，删除本文件。
- 禁止把本文件加入长期规范索引。

## 字段契约盘点与结果基准

范围：基础信息、配方、可售产品、价格、邮费、商家、客户、系统和对应文件输出。

- 为每个业务数值字段记录数据库类型、Java 类型、前端类型、单位或币种、输入精度、展示精度、计算用途和快照用途。
- 将字段分类为 ID/排序、计件数量、计量数量、尺寸、来源单价、交易单价、金额、比例、公式变量或技术指标。
- 标记字段是否进入公式、规则匹配、BOM、价格、折扣或历史快照。
- 记录现有接口请求响应、公式样例和输入方式；计算结果按新规范重新建立基准。
- 只有确认字段契约后才能替换组件或格式器。
- 禁止根据字段名、数据库 scale 或搜索结果直接批量修改。

## 公共数字组件与格式化能力

范围：`admin-ui/src/components`、`admin-ui/src/utils`、VXE Grid formatter/editor。

- 新增 `BusinessNumberInput`、`BusinessNumberText`、`BusinessInchInput`。
- 新增 `formatQuantity`、`formatUnitPrice`、`formatMoney`、`formatInch`、`formatRate`。
- 对现有 Element Plus 封装和 AutoNumeric 适配分别做最小验证，按原值保真、VXE 编辑兼容、中文输入、粘贴、键盘、可访问性和包体积选择实现。
- 禁止为一个数字控件引入第二套 UI 组件库。
- 支持 `COUNT`、`QUANTITY`、`UNIT_PRICE`、`MONEY`、`RATE` 业务模式。
- 支持最小/最大小数位、单位/币种精度、步长、范围、负数、清空、粘贴和键盘输入。
- `COUNT` 使用 0 位；`QUANTITY/UNIT_PRICE/RATE` 展示保底 2 位，并只删除第 2 位之后的尾零；`MONEY` 固定币种精度。
- 编辑态使用十进制文本保留原值，不强制补零；失焦态按业务格式展示；禁止格式化结果回写原值。
- 兼容现有 `number/string/null`、表单校验和 API payload，不改变已有接口字段类型。
- `BusinessInchInput` 固定由整数输入框和分数下拉组成，默认 `1/8 in` 选项为 `0`、`1/8`、`1/4`、`3/8`、`1/2`、`5/8`、`3/4`、`7/8`。
- 产品或配方配置其他步长时，由 `BusinessInchInput` 统一生成选项；页面禁止自行传入选项列表。
- `BusinessInchInput` 必须完成十进制值与“整数 + 分数”的双向无损转换。
- Grid 只读单元格使用轻量 formatter，编辑单元格使用统一 editor。
- 为 `1`、`0.5`、`0.325`、`1.2300`、`1.2304`、`0.0156`、`72.125`、负数和空值补测试。

## 代码生成器与 Demo

范围：`bocoo-modules-generator/src/main/resources/vm`、`bocoo-modules-demo`、前端 Grid 配置模板。

- 数量、单价、金额、比例和尺寸字段必须生成对应业务模式。
- 禁止生成散落的 `precision=2`、`toFixed(2)` 和未知语义 number formatter。
- 普通 integer、排序号、ID 和业务小数必须区分。
- Generator 与 Demo 必须使用相同数字组件和 formatter。
- 补生成产物检查，防止新模块重新引入旧写法。

## 基础信息

范围：产品分类、产品字典、单位管理、厂家管理、物料类型、物料属性、物料管理、资料资产及通用 `ProductEntityGridPage`。

- 产品分类、字典、厂家、物料类型和资料资产的 ID、排序号、文件大小必须保持整数语义。
- 单位管理必须统一 `precision_scale`、`rounding_mode`、单位步长和换算率。
- 单位舍入枚举必须支持并准确翻译 `HALF_UP`、`HALF_EVEN`、`CEILING`、`FLOOR`、`UP`、`DOWN`。
- 物料属性数值必须按属性单位精度录入和展示。
- 物料重量、门幅、厚度、单价和销售价必须按各自业务类型处理。
- `ProductEntityGridPage` 禁止用一个通用两位小数规则处理所有 number 字段。
- 基础信息列表、详情、选择器、导出和变更记录必须显示一致。

## 配方管理

范围：配方主表、原料池、配置项、限制规则、内部变量、用量规则、模拟、审核和版本快照。

- 配方宽度、高度和长度必须统一使用英寸分数控件。
- 原料池损耗率、内部变量、固定用量和最小/最大用量的录入、保存和回显必须对齐字段精度。
- 用量展示必须读取物料单位精度。
- 配方模拟、审核摘要和版本快照必须保持相同原值。
- 扫描配方页面中的 `precision=2` 和 `toFixed(2)`，只替换与字段契约不一致的录入和展示；USD 正式金额等合法两位边界必须保留。
- `72 1/8 in` 往返后必须保持 `72.1250`。
- 公式录入方式、变量、中文别名、函数名、条件配置和保存文本必须保持不变。

## 可售产品

范围：`SaleProductPage.vue`、可售产品 BO、VO、Entity、Service 和快照。

- 最小/最大宽高必须使用统一英寸分数格式。
- 配方版本中的尺寸、单位和价格引用必须保持原精度。
- 列表、详情和产品目录必须使用同一展示规则。
- 禁止可售产品页面自行固定两位小数。

## 价格设置与邮费模板

范围：价格设置、物料价格、批量价格、价格规则、价格表达式、邮费模板和邮费匹配。

- 物料单价、价格规则单价必须完整支持 4 位。
- `0.0156` 禁止被截断或展示为 `0.02`。
- 进入价格公式的物料单价必须保持 4 位；价格公式语义和正式金额舍入保持不变。
- 面积区间必须支持字段保存精度或面积单位精度。
- 邮费规则金额必须支持 4 位录入和计算，最终邮费按币种精度冻结。
- 邮费区间边界和临界小数必须补测试。
- 补 `0.0156 × 1000 = 15.60` 测试。

## 商家管理与商家等级

范围：商家档案、商家等级、等级折扣、授信额度及平台商家页面。

- 等级折扣必须明确保存倍率还是百分数，并保持现有 `numeric(10,4)` 契约。
- 折扣页面展示不得混淆 `0.95` 与 `95%`。
- 授信额度、已用额度、可用额度必须按币种精度处理。
- 等级门槛、排序号、商家数量和统计数量必须区分金额与整数语义。
- 商家详情、等级列表和折扣配置必须使用同一折扣规则。
- 补折扣临界值、累计金额和负数非法输入测试。

## 客户管理

范围：客户档案、平台客户、客户历史、信用额度和客户统计。

- 客户信用额度和累计消费金额必须按币种精度处理。
- 客户数量、历史交易数量和排序号必须保持整数语义。
- 客户列表、详情和业务选择器必须使用相同金额格式。
- 禁止客户页面自行使用 `toFixed(2)`。

## 公式计算引擎统一

范围：`ProductFormulaExpressionParser`、配方变量、用量计算、模拟、价格表达式、规则条件和相关测试。

- 保留当前 DSL 变量、中文别名、运算符、`round/ceil/floor` 函数名、条件配置和公式文本。
- 将后端公式数值执行从 `double` 统一为 `BigDecimal`，除法使用 `MathContext.DECIMAL128`。
- `round/ceil/floor` 分别使用 `HALF_UP/CEILING/FLOOR`，继续支持精度参数。
- 数值条件使用十进制比较；字符串、配置项和物料属性条件语义保持不变。
- `ProductFormulaSimulationUsageCalculator` 按用量规则、配方原料和物料主单位解析计算单位与舍入模式，最终使用计算单位精度，删除全局固定 `setScale(2)`。
- 价格表达式使用完整物料单价和用量，正式金额继续在 USD 2 位边界舍入。
- 前端保留 jsep 语法和原编辑方式；即时样例不得覆盖后端权威结果。
- 补现有公式文本兼容测试、正负临界值、除法、函数精度、条件比较、BOM 用量和价格试算测试。

## 数据库与 API 契约

范围：本轮业务对应的 PostgreSQL 初始化 SQL、Entity、BO、VO、Mapper、JSON 快照和 API 类型。

- 扫描全部 `numeric/decimal/integer/real/double precision` 字段。
- 物料与规则单价保持 4 位；正式 USD 金额保持 2 位。
- 配方数量和变量按现有 6 位字段契约；商家折扣按现有 4 位契约；损耗率按现有 6 位契约。
- 成品计件数量允许 integer，原料用量不得使用 integer。
- API、数据库和前端类型必须保持相同业务语义和精度。
- 本轮业务 JSON 快照必须保存原始数量、单价、金额、单位、币种和舍入结果。
- 禁止仅为统一外观批量修改数据库 scale。
- 数据库调整必须在对应整改项中列出影响和验证范围。
- 初始化 SQL 只保留最新结构；整改完成后清理并重新生成测试数据和快照。

## 系统、监控与 AI 费用

范围：AI 模型、AI 配额、AI 用量费用、监控指标、Dashboard 和统计页面。

- AI 单价和费用必须区分高精度单价与币种金额。
- Token 数、调用次数、缓存数量、字节数和时长必须保持整数或明确单位。
- 百分比、命中率和利用率必须使用比例格式器。
- Dashboard 汇总值必须与来源业务金额和数量一致。
- 系统页面禁止复用业务金额格式处理普通技术指标。

## 导入、导出与文件输出

范围：本轮业务已有的 Excel 导入导出、PDF、打印、邮件和文件快照。

- Excel 数值必须保持数字类型，禁止全部转成格式化字符串。
- 导入必须按字段业务类型校验精度和步长。
- 数量、单价、金额、比例和英寸必须使用各自格式。
- PDF、打印、邮件和页面必须复用同一格式规则。
- 历史文件必须读取冻结快照，禁止按最新基础数据重新计算。
