# 前端代码规范

## 1. 页面结构

普通管理页面优先对齐紧凑型 ERP 后台标准。视觉、密度、导航、Tab、按钮、表格和分页以 [admin-ui-visual-standards.md](./admin-ui-visual-standards.md) 为准，不再沿用默认组件拼装、大卡片、大胶囊或大留白风格。

```text
app-container
  -> el-form inline 查询区
  -> el-row toolbar + right-toolbar
  -> el-table / tree-table
  -> pagination
  -> el-drawer 表单，短操作可用 el-dialog
```

必须遵守：

- 页面目录优先放在 `admin-ui/src/pages/<domain>/`。
- API 封装放在 `admin-ui/src/api/<domain>/`，页面里不直接写请求 URL。
- 动态路由组件必须补到 `admin-ui/src/stores/permission.ts` 映射。
- 菜单、按钮、表头、placeholder、tooltip、空状态、错误提示使用 i18n key。
- 按钮使用 `v-hasPermi`，权限码和后端 `@SaCheckPermission`、数据库 `sys_menu.perms` 一致。
- 时间展示使用 `formatUtc()`；日期范围提交使用 `withUtcDateRange()` / `withUtcDateRangeParams()`。
- 标准化按业务场景决定，不按组件形态一刀切：单主体自维护页可用卡片表单，多记录资料维护页优先 grid，复杂录入页按业务工作流定制。

## 2. Grid 页面

- 查询区默认保持紧凑。字段多时按业务优先级收缩字段宽度或确认是否保留字段，不能通过隐藏滚动条掩盖布局问题。
- 查询、重置、工具栏按钮、表格行高、表头、分页密度必须对齐紧凑 ERP 视觉标准。
- 表格使用 `el-table`，批量操作需要选择列，长文本使用 `show-overflow-tooltip`。
- 普通 CRUD/grid 默认使用 `el-table`；不要因为项目已引入 `vxe-table` 就把普通资料维护页改成 vxe。
- `vxe-table` 只用于多行录入、单元格编辑、列拖动、键盘录入、批量粘贴、采购单/明细行这类复杂表格。
- 使用 `vxe-table` 时必须有稳定 `rowKey`、明确的单元格校验、保存 payload 规则和权限入口；远程选项仍通过 API 封装加载。
- 标准列顺序：选择列、序号列、业务列、操作列；没有行级业务动作的页面不要硬保留空操作列。
- 序号按分页偏移显示，不让用户自己数行。
- 操作列固定右侧，优先使用 `AdminTableActions`，不要在页面里重复手写行按钮、tooltip、更多菜单样式。
- 同一系统只保留一套表格行操作风格，禁止同页或跨页混用胶囊按钮和 Element Plus link 按钮。
- 行操作按语义决定展示：两个以内的动作全部外露；超过两个动作时只外露真正主操作，其余放入“更多”。
- 页面工具栏的颜色和层级统一由视觉规范控制：同一区域最多一个实心主按钮，普通修改/查看/导入/导出使用中性描边，危险动作使用弱化风险色。
- 主操作通常是用户最常走的正向动作，例如修改、详情、设置配方；引用检查、变更记录、删除、清空、分配用户、数据权限等不要默认当主操作。
- 删除、清空、强退、超级修改等高风险动作必须弱化或进入“更多”，并保持确认提示。
- 同一张表如果只有部分行无可用动作，可以保留操作列并在无动作行显示 `-`；如果当前业务列表整体无行级动作，应隐藏操作列。
- 常用操作顺序建议：主操作、辅助查看/检查、管理子资料、危险操作。
- 分页使用项目现有 `pagination` 组件。
- 普通分页 grid 的表头排序必须走服务端排序，使用 `orderByColumn` / `isAsc` 请求参数；禁止只对当前页数组排序。
- 只有字段配置显式声明可排序的列才显示排序入口，状态、附件、备注、JSON、长文本和操作列默认不开放排序。
- 普通 grid 支持双击行打开详情抽屉。
- 业务状态使用开关或明确状态标签，不裸显示 `true/false`、`1/0`、`ENABLED/DISABLED`。
- Boolean 业务字段展示为“是/否”，表单里用 `el-switch`。
- 普通 grid 需要整行强调时，业务页使用 `ProductGridConfig.rowTone` 声明语义，不在业务页面手写行背景色或复制 CSS class。
- `rowTone` 只表达业务风险或状态语义：`danger` 用于驳回、失败、严重异常；`warning` 用于待处理、未校验、风险提示；`success` 用于已通过、已启用、生效；`muted` 用于停用、作废、归档。
- 当前行和勾选行的选中态必须优先于 `rowTone`；普通启用/生效记录不默认整行着色。
- `rowClassName` 仅作为底层兼容和特殊扩展保留；普通业务状态优先使用 `rowTone`。

## 3. Drawer 和 Dialog

- 新增、编辑、详情、引用检查、复杂预览默认使用右侧抽屉，后台标准抽屉优先使用 `AdminDrawer`。
- 选择器、导入、批量操作、短信息查看等轻量弹窗优先使用 `AdminDialog`，底部操作区使用 `AdminDialogFooter`。
- 不在页面里重复覆写 `el-drawer` / `el-dialog` 的头部、底部、圆角、阴影和间距；确有特殊业务布局时只覆写内容区。
- 抽屉表单要有清晰分组和底部 sticky 操作栏，头部和底部高度保持紧凑，不留大面积空白。
- 字段多的页面抽屉宽度建议不小于页面宽度的 80%。
- 新增、编辑、超级修改、批量录入等会产生用户输入的抽屉必须接入未保存内容保护，优先复用 `useFormCloseGuard` / `useUnsavedChangesGuard`。
- 有用户输入的抽屉不要允许误点遮罩直接关闭；关闭按钮、取消按钮、Esc 都应先判断是否存在未保存变更。
- 详情、引用检查、只读预览可以直接关闭，不需要未保存保护。
- 查看界面不能只是把输入框禁用；应按只读文档、描述列表或表格化信息重排，适合打印/复核的内容优先使用详情布局。
- 删除、启停、强退、清空、同步确认使用 `ElMessageBox.confirm`。
- 审核原因、密码重置、少量输入可以使用小弹窗或 `ElMessageBox.prompt`。
- 图片预览、导入表选择、批量粘贴预览可以使用轻量 dialog。
- 配置编辑器、价格矩阵、发布闸门等复杂页面允许自定义内容区，但不能重做后台外壳。

## 4. 动态表单和自定义字段

- 动态属性、类型属性、自定义参数等字段应按业务录入习惯分组展示。
- label 和输入控件之间要保持稳定间距，不让文字贴住输入框。
- 数字值带单位时，数字输入和单位选择应同排展示；单位下拉不要独占整行。
- 单位下拉展示业务名称，提交稳定编码。
- 文本、数字、布尔、枚举等类型要用对应控件，不用一个文本框承载所有值。
- 动态字段如果来自后端模板，页面不在渲染过程中反写 form，避免触发误判为未保存变更。
- 展示型汇总字段由后端生成时，页面不要提供重复录入框。

## 5. 组件和 TypeScript

- 页面 SFC 只负责路由级编排、查询条件、数据加载和动作分发。
- 复杂业务区块优先复用公共组件；没有现成组件时按领域沉淀新组件。
- 通用组件放到 `admin-ui/src/components/` 或现有公共组件目录。
- 只服务单一业务域的组件放到 `admin-ui/src/pages/<domain>/components/`。
- 被两个以上页面复用时再上提为公共组件。
- 组件抽取以可独立理解、可复用或明显降低页面复杂度为准，不为了拆而拆。
- 新增或迁移页面使用 Vue SFC + TypeScript，API 返回值和表单模型要有显式类型。
- 页面文案通过 i18n key 和组件状态管理，不使用 DOM 文案 patch、全局桥接或页面级硬编码翻译。

## 6. 选择器和业务字典

- 下拉选择默认支持模糊过滤。
- 数据量可能增长的选择使用远程搜索、分页或弹出选择器。
- 产品配置中心单位下拉以业务单位表 `pc_unit` 为权威，不使用系统字典 `sys_dict_data.product_unit`。
- 产品业务枚举以下列业务表为权威：`pc_product_dict_type` / `pc_product_dict_item`。
- `sys_dict_*` 只保留后台框架和系统模块自己的底层字典，不承载产品业务枚举。

## 7. 前端验证

新增或调整页面后至少检查：

- `npm run typecheck` 或项目对应 typecheck 命令。
- 页面菜单可进入，刷新不 404。
- 搜索、重置、分页正常。
- 按钮权限可见性符合角色授权。
- 控制台无明显错误。
- 涉及时间展示时，用非中国时区抽查 UTC 显示。
