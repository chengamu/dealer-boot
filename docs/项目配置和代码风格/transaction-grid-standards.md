# 交易单据 Grid 标准

本文适用于报价、订单、支付、生产、发货、物流、应收等交易单据列表。基础信息、配置主数据和纯任务队列不直接套用全部规则。

## 页面定位

- 交易列表是业务事实的最终查询入口，不是只为工作台提供数据的附属页面。
- 每个仪表盘数字、待办和跨模块跳转必须落到一个稳定 Grid，并携带可解释的查询条件。
- 列表负责检索、查看和导航；修改状态、付款、生产、发货等动作回到所属详情或处理页执行。
- 同一业务对象可以复用 Grid 组件，但必须按登录角色裁剪查询字段、列、数据范围和动作。

## 默认时间

- 历史交易列表的第一个查询条件必须是该单据的主要业务时间。
- 默认范围为用户时区的当月第一天 `00:00` 至当天结束；请求边界按 UTC 规范转换。
- 点击重置恢复默认时间范围，不清空为无限历史。
- 导出、分页和仪表盘跳转必须沿用当前时间条件。
- 任务队列例外：待付款、待生产、待发货、支付异常等未完成队列默认展示全部有效待办，不得用当月范围隐藏跨月积压；页面要明确标识“全部待办”。

## 默认排序

- 历史交易列表按主要业务发生时间倒序，再按主键倒序，保证同一时间下顺序稳定。
- 不使用 `update_time` 作为交易列表默认排序，否则旧单据因状态同步或备注更新跳到顶部。
- 示例：销售订单使用 `submitted_time DESC, sales_document_id DESC`；支付单按创建或支付成功时间取决于页面任务；包裹查询按创建或发货时间取决于当前队列。
- 待办队列优先按风险和等待时长排序，例如到期时间升序、创建时间升序，不强行套用历史倒序。

## 角色筛选

- 商家和商家业务员只查询当前经营主体，不展示“商家”条件。
- 平台跨主体查询优先展示“经营主体”，不默认提供全局客户自由文本搜索。
- 同一经营主体存在多个账套时，平台销售、平台财务和管理报表必须支持按账套查询；只有一个账套时可隐藏条件并自动限定。
- 账套下拉只加载当前用户已授权账套，不允许用自由文本伪造账套范围。
- 平台确需查客户时，先选经营主体，再加载该主体客户选项，避免名称歧义和越权联想。
- 工厂生产以生产阶段、订单号、经营主体和产品信息为主，不展示报价草稿条件。
- 发货人员以发货阶段、订单号、经营主体、包裹号和运单号为主。
- 查询字段没有权限或对当前角色没有意义时直接隐藏，不用禁用控件占位。

## 查询区布局

- 可见查询区只保留一行，禁止换行堆叠成多行表单。
- 存在时间范围查询时，时间必须放第一位；其后只保留单号、主状态和当前角色最常用的对象条件。
- 一行放不下时，优先移除低频条件，不增加第二行，也不把控件压缩到不可读。后端 BO 和接口可保留低频查询能力，供仪表盘跳转、导出或后续扩展使用。
- 控件宽度按内容分配：时间范围和客户等长值可较宽，状态、类型等短选项使用窄下拉。所有下拉必须完整显示当前值和常见选项，禁止只剩箭头。
- 查询和重置按钮与条件保持同一行；重置同时恢复角色默认范围和默认日期。

## Grid 字段

- 列表至少包含业务单号、核心关联对象、关键金额或数量、主状态、独立阶段状态、主要业务时间和操作。
- 状态必须展示业务中文或当前页面语言，不直接展示枚举编码。
- 时间显示到分钟；金额、尺寸和数量遵守数值精度规范。
- 订单等跨阶段单据应分别展示订单、付款、生产、发货状态，禁止用一个含义过载的状态文本替代事实。
- 操作列固定右侧，仅展示当前权限和状态可执行的动作。

## 数据与权限

- 前端隐藏字段不能替代后端数据范围；查询服务必须按当前经营主体、角色、负责人或团队范围过滤。
- 当单据含账套归属时，后端还必须校验账套授权范围，不得因经营主体相同而默认可查全部账套。
- 平台跨主体接口不得把 `ignoreTenant` 等同于查看全部，必须叠加经营主体授权范围。
- 列表总数、分页数据、导出和仪表盘聚合必须使用同一数据范围语义。
- 每个查询页应在业务文档中记录主要时间字段、默认排序、角色筛选矩阵和详情入口。

## 验收

- 默认打开历史列表时日期为当月第一天至当天。
- 重置后日期和角色范围恢复默认值。
- 默认排序符合主要业务时间，不因普通更新时间改变。
- 不同角色看不到无意义或越权的筛选字段。
- 所有下拉值可读，Grid 无不必要的内部纵向滚动条。
- 仪表盘和待办可以跳入列表并准确还原筛选条件。
- 查询、分页、导出和详情返回使用一致的数据范围。
