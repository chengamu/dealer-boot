# 数值精度、展示与舍入规范

本规范适用于基础信息、配方、价格、报价、订单、支付、生产、物流、Excel 和 PDF。所有新增和重构功能必须遵守。

## 强制原则

- 每个数值字段必须明确：业务类型、单位或币种、录入精度、保存精度、计算精度、展示精度、舍入模式和舍入时点。
- 录入精度、保存精度、计算精度和展示精度必须独立定义，禁止用一个 `precision` 参数同时代替。
- 格式化只能改变展示，禁止改写原值。
- 物理数量必须读取 `pc_unit.precision_scale` 和 `pc_unit.rounding_mode`。
- 货币金额必须读取 `sys_currency.decimal_places`；USD 使用 2 位小数。
- 单价与金额必须分开处理。物料和价格规则单价允许 4 位，报价和订单成交单价按交易字段契约保存。
- 持久化数值和后端权威计算必须使用 `BigDecimal`。
- 公式录入契约必须保持稳定；公式计算内核必须按本规范统一。
- 页面、Excel、PDF、邮件和支付必须使用同一套数值规则。

## 精度定义

| 类型 | 强制定义 |
| --- | --- |
| 录入精度 | 用户允许输入的小数位和业务步长，由字段、单位、产品或规则确定 |
| 保存精度 | 数据库、API 和快照能够无损保存的位数 |
| 计算精度 | 计算中间过程和最终业务边界使用的精度与舍入方式 |
| 展示精度 | 页面和文件的显示格式，不得改变保存值和计算值 |

## 精度矩阵

| 数据类型 | 录入精度 | 保存精度 | 计算精度 | 展示规则 |
| --- | --- | --- | --- | --- |
| 计件数量，如个、只、套、对 | 0 位，必须为整数 | 通用数量使用 `numeric(18,6)`；只表达成品件数时允许 integer | 不额外舍入 | 0 位，例如 `1` |
| 计量数量，如 m、m²、kg | 最多 6 位，并受单位精度限制 | `numeric(18,6)` | 6 位 | 固定保底 2 位；第 3 位以后显示到最后一个非零位，例如 `1.00`、`0.50`、`0.325` |
| 换算率、持久化公式变量 | 最多 6 位 | `numeric(18,6)` | 十进制计算，中间过程不固定截断 | 最多 6 位，去除无意义尾零 |
| 英寸宽度、高度、长度 | 按产品步长录入，默认 `1/8 in` | 默认 `numeric(18,4)` | 6 位 | 使用分数英寸，例如 `72 1/8 in` |
| 公制尺寸、面积、重量 | 最多 6 位，并受单位精度限制 | `numeric(18,6)` | 6 位 | 固定保底 2 位；第 3 位以后显示到最后一个非零位 |
| 物料单价、价格规则单价 | 最多 4 位 | `numeric(18,4)` | 4 位 | 最少 2 位、最多 4 位 |
| 报价、订单产品单价 | 不得超过交易字段精度 | USD 使用 `numeric(18,2)` | 在交易金额边界舍入 | USD 固定 2 位 |
| 行金额、折扣、邮费、税额、付款、总额 | 不得超过币种精度 | USD 使用 `numeric(18,2)` | 在金额边界舍入 | USD 固定 2 位 |
| 折扣率、损耗率及其他比例 | 不得超过字段契约精度 | 商家折扣使用 `numeric(10,4)`；配方损耗率使用 `numeric(10,6)` | 使用字段保存精度 | 固定保底 2 位；第 3 位以后显示到最后一个非零位；API 必须明确保存倍率还是百分数 |

## 数量

- 成品数量和原料用量必须使用不同字段语义。报价、订单中的成品件数按销售单位校验；配方、BOM 和生产耗用按物料单位校验。
- 计件单位的 `precision_scale` 必须为 `0`。
- 计量单位的 `precision_scale` 必须按业务设置为 `1` 至 `6`。
- 通用数量字段必须使用 `numeric(18,6)`。
- `precision_scale` 只表示最大小数位，不表示业务步长。整卷、整包或每 `0.125` 递增必须由单位、产品或业务规则明确配置。
- 输入值必须满足单位精度和单位步长。
- `1.5 PCS` 必须阻止保存，禁止静默舍入为整数。
- 计量数量展示规则必须为：

```text
minimumFractionDigits = 2
maximumFractionDigits = max(2, unit.precisionScale)
```

- `1 m` 必须显示为 `1.00 m`。
- `0.5 m` 必须显示为 `0.50 m`。
- `0.325 m` 必须显示为 `0.325 m`，禁止显示为 `0.33 m`。
- `1.2300 m` 必须显示为 `1.23 m`。
- `1.2304 m` 必须显示为 `1.2304 m`。

## 英寸尺寸

- 订单宽度、高度和长度必须复用 `BusinessInchInput`。
- `BusinessInchInput` 内部必须固定由“整数英寸输入框 + 分数下拉选择器”两个控件组成。
- 整数英寸输入框只能录入非负整数；分数禁止通过自由文本录入。
- 未配置产品尺寸步长时必须使用 `1/8 in`。
- `1/8 in` 对应选项必须为：`0`、`1/8`、`1/4`、`3/8`、`1/2`、`5/8`、`3/4`、`7/8`。
- 产品或配方明确配置其他尺寸步长时，`BusinessInchInput` 必须根据步长统一生成分数选项。
- 页面禁止自行传入、拼接或修改分数选项。
- 保存时必须转换为十进制，例如 `72 1/8 in` 保存为 `72.1250`。
- 页面、Excel、PDF 和生产文件必须恢复为分数英寸展示。
- 禁止把 `72.125` 显示为 `72.13`。
- 尺寸取整方向必须由产品和安装规则明确指定，禁止页面推断。

## 单价

- 物料单价和价格规则单价必须支持 4 位小数。
- 物料单价必须使用完整 4 位进入既有价格公式，禁止在进入公式前压缩为 2 位。
- 物料单价展示必须最少 2 位、最多 4 位，并只去除第 2 位之后的尾零。
- 报价和订单产品单价属于冻结交易金额，USD 必须保存并显示 2 位。
- `5` 必须显示为 `$5.00`。
- `0.5` 必须显示为 `$0.50`。
- `0.325` 必须显示为 `$0.325`。
- `0.0156` 必须显示为 `$0.0156`。
- 禁止以 `precision=2` 接收或编辑 `numeric(18,4)` 单价字段。

## 公式计算兼容

- 配方 DSL 和价格 DSL 的录入方式属于稳定业务接口。
- 变量、中文别名、运算符、函数名、括号、条件表达式和公式保存文本必须保持兼容。
- 技术科已有公式必须能够继续录入、保存、校验和执行。
- 后端公式计算必须使用十进制数值模型，禁止通过二进制浮点执行权威结果。
- 加、减、乘必须保留完整中间精度；除法必须使用 `MathContext.DECIMAL128`，禁止固定截断中间结果。
- `round(value, scale)` 必须使用 `HALF_UP`。
- `ceil(value, scale)` 必须使用 `CEILING`。
- `floor(value, scale)` 必须使用 `FLOOR`。
- 数值条件比较必须使用十进制比较，字符串和配置项条件语义保持不变。
- 最终物料用量的计算单位必须按“用量规则 `calculation_unit_code` -> 配方原料 `calculation_unit_code` -> 物料 `unit_code`”解析。
- 最终物料用量的舍入模式必须按“用量规则 `rounding_mode` -> 配方原料 `rounding_mode` -> 计算单位 `rounding_mode`”解析。
- 最终物料用量必须使用计算单位 `precision_scale`，禁止全局固定 2 位。
- 价格公式必须使用完整物料单价、用量和尺寸；正式金额只在币种边界舍入。
- 前端即时校验、后端校验、配方模拟和价格试算必须使用相同语法契约；后端结果是权威结果。
- 公式计算内核调整后必须清理测试配方、版本、报价和订单快照并重新生成。

## 金额

- USD 行金额、折扣、邮费、税额、付款金额和总额必须保存并显示 2 位小数。
- 其他币种必须读取 `sys_currency.decimal_places`，禁止写死 2 位。
- 支付接口必须按币种最小单位传输；USD `$10.00` 必须传输为 `1000` cents。
- 金额计算顺序必须为：
  1. 使用完整数量、尺寸、单价和公式结果计算。
  2. 形成正式行金额时按币种精度舍入。
  3. 折扣、邮费和税额分别按币种精度冻结。
  4. 单据总额累加冻结后的正式金额。
- 页面、Excel、PDF、邮件和支付金额必须完全一致。

## 舍入

| 业务类型 | 强制规则 |
| --- | --- |
| 尺寸、数量、条件和规则边界 | 使用未格式化的十进制原值比较，禁止使用页面显示值判定 |
| 测量、检验、极限值判定 | 使用符合 GB/T 8170 的偶数修约；禁止连续修约 |
| 报价、订单、折扣、邮费、付款 | 在最终金额边界使用 `HALF_UP` |
| 生产用量、单位换算 | 使用单位、配方或用量规则指定的 `rounding_mode` |
| 整包采购、必须向上补足 | 业务规则必须明确指定 `CEILING` |
| 整件、整套数量 | 校验单位步长，禁止通过舍入修正非法输入 |
| 公式 DSL `round/ceil/floor` | 分别使用 `HALF_UP/CEILING/FLOOR`，精度由函数第二参数指定 |

- `HALF_UP`、`HALF_EVEN`、`CEILING`、`FLOOR`、`UP`、`DOWN` 必须按 Java `RoundingMode` 语义映射。
- `UP` 表示远离零，`DOWN` 表示向零；禁止将其错误解释为数学上的向上或向下取整。
- `BigDecimal.setScale()` 必须显式传入 `RoundingMode`。
- 禁止将 `toFixed()`、格式化字符串或前端浮点结果用于权威计算。
- 禁止对同一个值连续舍入。

## 前端

- 必须复用统一格式化函数：`formatQuantity`、`formatUnitPrice`、`formatMoney`、`formatInch`、`formatRate`。
- 数字录入必须复用 `BusinessNumberInput`；禁止业务页面直接配置 `el-input-number` 精度。
- 数字只读展示必须复用 `BusinessNumberText` 或 Grid 数字 formatter；禁止页面自行拼接尾零和单位。
- 英寸尺寸录入必须复用 `BusinessInchInput`。
- `BusinessNumberInput` 必须支持 `COUNT`、`QUANTITY`、`UNIT_PRICE`、`MONEY`、`RATE` 五种业务模式。
- 数字组件必须支持：最小小数位、最大小数位、单位精度、币种精度、最小值、最大值、步长、负数控制、只读、禁用、清空、粘贴和键盘录入。
- `COUNT` 必须录入和展示 0 位小数。
- `QUANTITY` 必须允许录入到单位精度，展示时保底 2 位，并删除第 2 位之后的尾零。
- `UNIT_PRICE` 必须允许录入 4 位，展示时保底 2 位，并删除第 2 位之后的尾零。
- `MONEY` 必须按币种精度固定展示，不得删除币种要求的小数位。
- `RATE` 必须允许录入到字段精度，展示时保底 2 位，并删除第 2 位之后的尾零。
- 数字组件必须保留合法原值，禁止通过格式化修改、补零或舍入业务值。
- 数字组件必须使用十进制文本保存编辑中的原值，禁止在输入过程中通过 `parseFloat`、`Number` 或 `toFixed` 改写原值。
- 数字组件必须兼容现有字段的 `number/string/null` 输入输出和 API payload，禁止在批量替换中改变接口类型。
- 数字组件获得焦点时必须显示完整可编辑原值，禁止强制补零；失焦时必须按业务模式格式化展示。
- 表单失焦展示、详情展示、Grid 展示、Excel 和 PDF 必须遵循相同的保底位数和尾零规则。
- Grid 只读单元格必须使用轻量 formatter，禁止为每个单元格挂载输入组件。
- 数字组件只能负责录入、解析、校验和展示，禁止承担权威业务计算。
- 禁止为数字格式化引入第二套 UI 组件库。
- 页面禁止自行使用 `toFixed(2)`、`precision=2` 或固定两位 `Intl.NumberFormat` 处理未知业务类型。
- 数量、尺寸、单价、金额和百分比必须右对齐。
- 表头必须标明单位或币种。
- 输入控件必须保留并校验完整精度。
- 展示格式化结果禁止回写业务字段。

## 后端

- 持久化业务小数必须使用 `numeric`，Java 领域模型必须使用 `BigDecimal`。
- 数量、尺寸、单价、金额和公式权威计算禁止使用 `float`、`double` 或 `doubleValue()`。
- 公式解析器内部数值必须使用 `BigDecimal`；字符串和布尔条件保持原类型。
- 计算方法必须在明确业务边界舍入。
- 发现 `doubleValue()`、`setScale(2)`、`toFixed(2)` 或 `precision=2` 时必须先判断字段语义；公式计算和用量边界按本规范修正，合法的交易金额两位边界必须保留。
- API 比例字段必须明确保存倍率还是百分数。
- 快照必须保存原始数量、原始单价、最终金额、单位、币种和舍入结果。
- 基础单位、币种或价格规则变更后，禁止重新解释历史报价和订单。

## Excel、PDF 与打印

- Excel 数值单元格必须保持数字类型。
- Excel 计件数量格式必须为 `0`。
- Excel 计量数量格式必须为 `0.00####`。
- Excel 单价格式必须最少 2 位、最多 4 位。
- Excel 金额必须按币种精度固定小数位。
- 分数英寸必须使用统一分数格式或稳定字符串输出。
- PDF、打印和邮件必须复用与页面相同的格式化规则。

## 验收清单

1. 每个数值字段是否明确业务类型、单位或币种？
2. 数据库保存精度是否满足本规范？
3. 页面输入是否截断数据库允许的精度？
4. 页面是否存在业务字段直接使用 `toFixed(2)` 或 `precision=2`？
5. 权威业务计算是否错误使用 `double` 或 `doubleValue()`？
6. 公式 DSL 是否保持原录入方式、变量、中文别名、函数名和保存文本？
7. `setScale()` 是否位于明确业务边界并显式指定 `RoundingMode`？
8. 页面、Excel、PDF、邮件和支付是否使用同一格式规则？
9. 单位、币种和数值快照是否完整？
10. 行金额与总金额在所有输出中是否完全一致？

## 测试基线

- 计件单位必须拒绝 `1.5`。
- 米单位必须接受并原样保存 `0.325`。
- `1 m`、`0.5 m`、`0.325 m` 必须分别显示为 `1.00 m`、`0.50 m`、`0.325 m`。
- `1.2300` 必须展示为 `1.23`，`1.2304` 必须展示为 `1.2304`。
- `0.0156 × 1000` 必须得到正式金额 `15.60`，禁止先把单价舍入为 `0.02`。
- `72 1/8 in` 往返保存后必须仍为 `72.1250`，并显示为 `72 1/8 in`。
- 页面、Excel、PDF、邮件和支付总额必须完全一致。
- 正数、负数和临界值必须覆盖 `HALF_UP`、`HALF_EVEN`、`CEILING` 和 `FLOOR`。
- 基础单位、币种或价格规则变更后，历史报价和订单结果必须保持不变。
