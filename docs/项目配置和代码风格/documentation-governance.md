# 文档治理规则

文档只保留能帮助后续开发、测试、业务维护和 AI 协作的稳定信息。执行计划、调试流水、截图证据和未确认想法留在对话或 Git 记录中。

## 入口

- `docs/README.md` 是唯一总入口。
- `docs/项目配置和代码风格/README.md` 是代码规范入口。
- `AGENTS.md` 只放协作启动规则，不复制专项规范。
- 没有被 `docs/README.md` 或对应目录 `README.md` 索引的文档，不作为当前开发依据。

## 放置

| 内容 | 位置 |
| --- | --- |
| 代码、权限、i18n、部署规则 | `docs/项目配置和代码风格/` |
| 产品配置业务知识 | `docs/产品配置知识库/` |
| 基础信息业务文档 | `docs/产品配置知识库/基础信息/` |
| 商家体系业务文档 | `docs/商家体系/` |
| 页面体验问题 | 对应功能目录下的 `体验问题清单.md` |

## 保留标准

文档必须至少满足一项：

- 描述当前功能如何使用。
- 描述当前接口、权限、字段、选择器、AI 槽位或数据契约。
- 描述当前代码规范、部署规则或文档规则。
- 记录仍需处理的页面体验问题。

不满足保留标准的内容应删除或合并。

## 更新规则

- 业务功能变化时，同步更新操作手册、接口与数据契约或体验问题清单。
- 规范变化时，更新本目录对应专项规范，不复制到 `AGENTS.md`。
- 测试发现的问题写入 `体验问题清单.md`，不写入稳定操作手册。
- 未确认的方案用 TODO 标记，并写明确认人或确认条件。

## 可检索性与引用关系

全局搜索、跨模块检索和大文档问答后续由独立 AI Gateway / 知识检索服务承载；当前业务后端仍以数据库为事实来源，负责权限、租户、业务校验、详情查询和审计。不要为了未来搜索能力提前接入 OpenSearch / Elasticsearch，也不要在业务模块中预铺搜索索引表。

新增或调整 P0/P1 业务功能时，必须在对应 `接口与数据契约.md` 判断是否需要记录可检索性与引用关系。满足以下任一条件时需要记录：

- 有稳定业务编码、单号、名称或摘要，后续可能被全局搜索命中。
- 会被其他模块引用，或会引用其他主数据、单据、文档。
- 搜索结果需要跳转到详情页、编辑页或审核页。
- 搜索结果需要按租户、状态、删除标记或权限过滤。

可检索性与引用关系只记录稳定业务口径，不设计搜索引擎实现。至少说明：

- 业务编码或单号。
- 搜索展示字段。
- 主要引用关系。
- 权限过滤依据。
- 详情入口。

核心业务值以业务数据库和详情接口为准。搜索索引以后只负责召回和展示摘要，不能作为库存数量、价格、金额、审核状态等关键业务事实的最终依据。

## 人工测试闭环

P0/P1 功能完成实现、修复或人工测试后，必须做文档影响判断。

1. 人工测试或浏览器验证功能行为。
2. 判断测试结果影响哪类文档：
   - 正确用法、录入顺序、常见问题变化：更新 `操作手册.md`。
   - 接口、权限、字段、选择器、AI 槽位、可检索性或引用关系变化：更新 `接口与数据契约.md`。
   - 页面体验、缺陷、待确认问题：更新 `体验问题清单.md`。
3. 执行覆盖本次变更的最小验证。
4. 回复中说明实际验证结果和文档更新结果。

## AI 代办知识库

业务功能使用双层文档结构：

- `操作手册.md` 写业务化 AI 说明：用户说法、追问场景、保存前确认摘要。
- `接口与数据契约.md` 写工具化 AI 契约：调用接口、权限、槽位、payload 边界、禁止暴露字段、失败提示。
- 领域总规则写在对应知识库的 `AI代办能力规范.md`。

P0/P1 功能如果需要 AI 代办，必须同时具备业务化说明和工具化契约；否则不能标记为 AI 可接入。
