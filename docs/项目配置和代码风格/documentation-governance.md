# 文档治理规则

## 目标

文档只保留能帮助后续开发、测试、业务维护和 AI 协作的稳定信息。一次性过程、历史方案、临时计划和未确认想法留在对话或 Git 历史里，不进入当前文档索引。

## 入口规则

- `docs/README.md` 是唯一总入口。
- 根目录 `README.md` 只放项目概览和入口，不承载业务细节。
- `AGENTS.md` 只放协作启动规则，不复制具体代码规范。
- 没有被 `docs/README.md` 或对应目录 `README.md` 索引的文档，不作为当前开发依据。

## 放置规则

| 内容 | 位置 |
| --- | --- |
| 长期代码规则 | `docs/项目配置和代码风格/` |
| 文档治理规则 | `docs/项目配置和代码风格/documentation-governance.md` |
| 基础信息业务文档 | `docs/产品配置知识库/基础信息/` |
| 基础信息体验问题 | 对应功能目录下的 `体验问题清单.md` |
| 待重写业务域目标 | 先在对话确认；确认后再建对应知识库 |

## 删除规则

满足任一条件时，应删除或合并，不继续保留在当前文档体系：

- 与当前代码实现不一致，且没有维护价值。
- 只是阶段性计划、执行记录、截图证据或调试流水。
- 与已有文档重复，只是换了目录或标题。
- 会让后续开发误以为旧方案仍可执行。
- 只用于追溯历史，Git 历史已经足够。

删除后不要在入口文档里反复列出被删目录；入口只描述当前可信内容。

## 更新规则

- 业务功能变化时，同步更新对应操作手册、接口与数据契约或体验问题清单。
- 规范变化时，更新本目录对应专项规范，不复制到 `AGENTS.md`。
- 测试发现的问题写入 `体验问题清单.md`，不要写入稳定操作手册。
- 未确认的方案用 TODO 标记，并写明需要谁确认；不要写成已定规则。

## 协作工作流

后续任务按“进入 -> 计划 -> 实现 -> 文档 -> 验证 -> 回复”闭环执行。

| 阶段 | 动作 | 文档规则 |
| --- | --- | --- |
| 任务进入 | 先读 `AGENTS.md`、`docs/README.md`，再按领域读对应入口 | 不新建临时计划文件 |
| 计划 | 复杂变更先确认目标、影响范围和验收方式 | 临时计划留在对话 |
| 实现 | 按 minimal diff 修改，避免无关重构 | 不把执行过程写入 docs |
| 文档更新 | 判断是否影响操作手册、接口契约、体验问题或规范 | 只写长期有效内容 |
| 验证 | 执行覆盖本次变更的最小检查 | 不把完整日志写入 docs |
| 回复 | 说明改了什么、验证了什么、未验证什么 | 不声称未执行的检查已通过 |

普通小修可以压缩计划和实现说明，但仍要做最小验证和文档影响判断。复杂业务重写必须先在对话中确认目标和边界，再创建或更新知识库。

## 人工测试闭环

P0/P1 功能完成实现、修复或人工测试后，必须做一次文档影响判断，不能只回复“已测试”。

固定流程：

1. 人工测试或浏览器验证功能行为。
2. 判断测试结果影响哪类文档：
   - 正确用法、录入顺序、常见问题变化：更新 `操作手册.md`。
   - 接口、权限、字段、选择器、AI 槽位变化：更新 `接口与数据契约.md`。
   - 页面体验、缺陷、待确认问题：更新 `体验问题清单.md`。
3. 执行能覆盖本次变更的最小验证。
4. 回复中说明实际验证结果，以及更新了哪些文档。

如果测试没有导致文档变化，也要在回复中说明“已判断无文档更新需求”和原因。

## AI 代办知识库

产品业务功能使用双层文档结构，不为每个功能新增独立 AI 文档：

- `操作手册.md` 写业务化 AI 说明：用户说法、追问场景、保存前确认摘要。
- `接口与数据契约.md` 写工具化 AI 契约：调用接口、权限、槽位、payload 边界、禁止暴露字段、失败提示。
- 总规则写在 `docs/产品配置知识库/AI代办能力规范.md`。

P0/P1 功能如果后续需要 AI 代办，必须同时具备业务化说明和工具化契约；否则不能标记为 AI 可接入。
