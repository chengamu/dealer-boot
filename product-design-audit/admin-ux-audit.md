## 步骤健康度

1. 侧边栏导航：中等。主分组可理解，但产品相关组过多且与基础/系统组混在同一层，非产品用户会被干扰。
2. 用户管理：较好。搜索、组织树、批量按钮、列设置、分页齐全，但按钮层级略拥挤。
3. 角色管理：较好。基础表格能力完整，但行操作只有图标，含义依赖经验。
4. 操作日志：中等。筛选维度完整，批量清空/导出齐全，但“系统模块”页面标题与“操作日志”业务语义不一致。
5. 代码生成：中等偏弱。空状态有显示，但缺少可执行下一步引导。
6. 经销商管理：较好。业务状态、等级、报价/订单数据集中，但筛选入口偏轻，批量操作不明显。
7. 订单详情：较好。订单状态时间轴清楚，付款状态与金额摘要醒目，但状态文案和操作按钮还可以更统一。
8. 新增角色弹窗：中等。功能完整，但树形权限密度偏高，弹窗内说明和分区还不够强。
9. 操作日志详情抽屉：中等。内容完整，但长 URL / 方法名 / JSON 参数可读性弱，容易显得粗糙。

## 主要优点

- 后台没有大面积混乱，基础骨架稳定：顶部栏、标签页、侧边栏、列表页、分页、右侧工具按钮已经形成统一框架。
- 系统用户、角色、日志这些高频管理页都有搜索、重置、批量选择、导出、分页等基础能力。
- 订单详情的状态链路很直观，用户能快速判断“已提交 -> 生产中 -> 已发货 -> 已送达”的进度。
- 经销商管理把指标卡和列表结合在一起，能看到待审批、活跃经销商、停用账号、VIP 经销商等运营状态。

## 轻量优化建议

1. 侧边栏保留现有结构，但给“产品相关”做降噪。
   - 为什么改：当前侧边栏里“基础信息 / 产品工程设计 / 产品配置中心 / 产品发布”与“系统管理 / 系统监控 / 代码生成 / 日志管理”同层排列。对非产品后台任务来说，导航认知负担偏高。
   - 怎么改：不删菜单，不重做导航。建议把产品相关组在非产品角色下默认收起，或在菜单排序上让“商家中心 / 商家管理 / 系统管理 / 系统监控 / 日志管理”形成连续区域；产品相关组保持原有入口但靠后。

2. 侧边栏一级菜单命名再明确一点。
   - 为什么改：“基础信息”在当前截图里容易被理解成系统基础资料，也可能被理解成产品基础资料，和产品配置边界不清。
   - 怎么改：如果它实际是产品基础资料，改成“产品基础信息”；如果是通用基础资料，保留“基础信息”但把产品相关子项移出或改到产品分组下。

3. 列表页顶部搜索区统一成“常用条件 + 更多筛选”。
   - 为什么改：用户管理、角色管理、操作日志的搜索区都完整，但横向铺开后视觉重量大，尤其操作日志有 6 个条件，占用首屏注意力。
   - 怎么改：保留 2-3 个高频字段直接展示，其余收进“更多筛选”；搜索和重置按钮固定在右侧，避免每页按钮位置漂移。

4. 批量操作按钮加选择反馈。
   - 为什么改：新增/修改/删除/导入/导出都有，但未选中时禁用按钮看起来像不可用功能，新用户不一定知道要先勾选。
   - 怎么改：在批量按钮旁增加轻量文本“已选 0 项”，勾选后变成“已选 N 项”；删除、修改只在选择后高亮。

5. 行操作图标需要 tooltip 或文字统一。
   - 为什么改：角色管理里的编辑、删除、授权、分配用户等是纯图标；用户管理又用更多菜单；日志管理用“详细”文字。三者理解成本不一致。
   - 怎么改：保持现有布局，给所有图标按钮补 `title/aria-label/tooltip`；同类表格统一成“更多”菜单或统一成文字按钮，不必大改样式。

6. 空状态要给下一步。
   - 为什么改：代码生成页显示“暂无数据”，但不知道是没导入表、数据源错误，还是没有权限。
   - 怎么改：空状态下显示一句原因型文案，例如“暂无可生成表，请先点击导入同步数据表”，并把“导入”作为主按钮。

7. 订单状态时间轴补充状态解释。
   - 为什么改：订单详情进度清楚，但“生产中 / 已发货 / 已送达”对业务人员还缺少预计责任方或下一动作。
   - 怎么改：在当前节点下增加一行小字，例如“工厂生产中，预计 Jun 12 发货”；不改变流程结构。

8. 订单详情的右侧摘要卡片增强层级。
   - 为什么改：金额、付款状态、付款方式都清楚，但卡片内部所有字段重量接近，高级感还差一点。
   - 怎么改：总金额作为唯一主视觉；付款状态使用统一状态色块；次要字段如付款时间、付款方式降低字号或颜色。

9. 表单弹窗增加分区说明和底部固定操作。
   - 为什么改：新增角色弹窗功能完整，但权限树很密集，用户容易直接面对一大坨勾选项。
   - 怎么改：保留弹窗结构，在“菜单权限”上方增加一句说明；底部按钮固定在抽屉/弹窗底部，滚动时仍可见。

10. 日志详情抽屉优化长文本可读性。
    - 为什么改：请求 URL、方法名、请求参数、返回参数都是长文本，当前直接铺在表单项里，不方便复制和定位。
    - 怎么改：长文本用等宽字体、浅底代码块、自动换行；JSON 参数支持折叠或至少加 `white-space: pre-wrap`。

11. 卡片圆角、阴影、边框统一收敛。
    - 为什么改：经销商管理卡片、订单卡片、系统表格容器的质感接近，但不同页面的阴影和边框强弱略有差异。
    - 怎么改：抽出轻量 CSS token 或复用现有变量，统一 `border-color`、`box-shadow`、`border-radius`，不用改组件结构。

12. 顶部标签页与面包屑语义再对齐。
    - 为什么改：操作日志页面面包屑显示“首页 / 系统模块”，标签显示“系统模块”，但实际页面是操作日志；会降低定位感。
    - 怎么改：检查菜单标题与路由 meta，让面包屑、标签页、侧边栏选中态都显示同一业务名。

## 可直接交给 Codex 的任务清单

1. 调整非产品后台菜单排序：让商家中心、商家管理、系统管理、系统监控、代码生成、日志管理连续展示；产品工程/产品配置/产品发布保持存在但靠后或默认收起。
2. 明确“基础信息”菜单命名：根据实际归属改为“产品基础信息”或调整到对应产品分组下。
3. 为 `admin-ui/src/layout/components/Sidebar` 的一级菜单折叠态与展开态补充 tooltip/可访问名称，保证图标导航可解释。
4. 在通用列表页批量工具条增加“已选 N 项”提示，优先覆盖用户管理、角色管理、操作日志。
5. 统一表格行操作：给角色管理图标按钮补 tooltip，并评估与用户管理“更多”菜单保持一致。
6. 优化代码生成空状态：暂无数据时展示“请先导入数据表”的说明，并突出导入按钮。
7. 修正操作日志路由标题：保证侧边栏、面包屑、标签页显示“操作日志”，不要显示“系统模块”。
8. 优化订单详情当前状态说明：在当前时间轴节点补充业务下一步说明，不改变状态流程。
9. 优化订单详情右侧摘要卡片：统一金额、付款状态、次要信息的字号、颜色和间距层级。
10. 优化新增角色弹窗：权限树上方增加说明文案，底部操作按钮保持固定可见。
11. 优化操作日志详情抽屉：长 URL、方法名、请求参数、返回参数使用浅底代码块与自动换行。
12. 提炼后台容器统一样式：统一卡片/表格容器的圆角、边框、阴影变量，先覆盖系统用户、角色、日志、经销商管理、订单详情。
